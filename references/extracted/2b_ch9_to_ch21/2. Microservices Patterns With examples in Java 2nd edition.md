to be a bottleneck.
RELIABLE
Just like tests, deployments should be reliable. Deployments should not fail
randomly, especially given their high frequency, because unreliable
deployments waste valuable time diagnosing and fixing issues. Automation
is one way to eliminate human error and significantly improve the reliability
of deployments. There’s also a design aspect, however.
Releasing a new version of a component typically involved terminating old
processes and starting new ones. But it should, however, not cause visible
failures. For example, the production environment should implement
connection draining to ensure that in-flight (HTTP) requests are completed
before the old instances are terminated. Additionally, long-running
background tasks, such as batch jobs that process large volumes of data,
should be designed to be interruptable and resumable, in order to not delay
upgrades or losing any work.

COMPONENT PER TEAM
Yet another important finding by the Accelerate authors is that high
performance teams can deploy their changes without coordinating with other
teams. One obvious way to promote team autonomy is through a multi-
component architecture, where each team has their own component with its
own deployment pipeline. This architecture enables a team to deploy their
changes independently, without needing to coordinate with other teams.
CONFIGURABLE FOR DIFFERENT ENVIRONMENTS
A key requirement is that a component can be deployed to multiple
environments without needing to be rebuilt each time. For example, the
deployment pipeline might deploy the component in a series of
environments, such as test and staging environment, before it’s finally
deployed to production. We should not need to rebuild the component for
each environment. Instead, as shown figure 5.19, a component should be
built once and then deployed to multiple environments.

INCREMENTAL/PROGRESSIVE DEPLOYMENTS
No matter how thoroughly we test our software, production bugs are
inevitable. Therefore, it’s essential to mitigate the impact of a buggy
deployment on users. One effective strategy is to decouple deployment—
running an updated component in production—from release—routing traffic
to the updated component. This approach allows for a progressive rollout,
where the updated component is gradually exposed to an increasing
percentage of users, thereby reducing potential disruptions.
For example, one common strategy is to do canary releases, where the
updated component is deployed alongside the current version. Initially, only
a small percentage of the traffic is routed to the updated component, with the
share gradually increasing until it handling 100% of the traffic. If at any
point, the telemetry (see observability below) from the updated component
indicates a problem, the deployment is rolled back. I’ll discuss incremental
releases in more detail in chapter {chapter-kubernetes}.
QUICKLY VERIFIABLE
Another key requirement is the ability to quickly determine whether a
deployment succeeded or failed. One very basic indicator of success is the
newly deployed component becoming ready to handle production traffic.
However, mere readiness does not guarantee that it actually works. The
component might, for example, be misconfigured and respond to every
HTTP request with an HTTP 500. Alternatively, an inefficient query might
cause the component to respond slowly.
Observability, which is discussed below, plays a central role in determining
whether a deployment was successful. The deployment mechanism should
analyze the new component’s telemetry - the result of handling traffic - to
determine if it’s running successfully. If a component is, for example,
exhibiting a high error rate or high latency, the deployment mechanism
should treat the deployment as having failed.

UNDOABLE
If a deployment fails, the deployment mechanism should be able to quickly
roll back the deployment. Ideally, rolling back should be as fast and as
simple as routing traffic back to the still running older component instances.
Alternatively, the deployment mechanism will need to restart the previous
version of the component, which typically takes longer and exacerbates the

Failed deployments must be quickly detected and rolled back

5.7 Summary
Fast flow requires an architecture that satisfies five quality attributes:
modifiability, evolvability, testability, deployability, and observability.
The first four attributes - modifiability, evolvability, testability, and
deployability - enable developers to rapidly deliver changes to
customers. The fifth attribute - observability - is important for two
reasons: it ensures the safety of deployments, and, it provides
developers with feedback from production and users.
Modifiability reflects how easily changes can be made to the
application. It is achieved through designing an architecture that is
loosely design-time coupled and cohesive.
Evolvability describes how easy it is to upgrade the application’s
technology stack. For large applications, it is achieved by using a multi-
component architecture which allows for most architectural decisions to
be made separately for each component and for upgrades to be done
incrementally.
Testability is a quality attribute that reflects how easy and fast it is to
verify that a change is releasable. It is achieved through designing an
architecture that is modular, loosely design-time coupled, and
deterministic.

---

## Page 188

Deployability is a quality attribute that reflects how easy and fast it is to
deploy a change into production. It is achieved through designing an
architecture that supports deployments that are automated, fast, reliable,
can be done without affecting SLAs, and can be automatically rolled
back when there’s a problem.
Observability reflects how easy it is to understand the behavior of the
application and its users. It is primarily achieved through designing an
architecture that implements the observability patterns. A multi-
component architecture can be used to improve the observability of
large and complex applications.
An application’s component architecture - one component or many -
influences how well a large application can satisfy the evolvability,
testability, deployability, and observability quality attributes, a key
differentiator between microservice and monolithic architectures.

6 The monolithic architecture
This chapter covers
The benefits and limitations of the monolithic architecture
How well a monolithic architecture satisfies the quality attributes for
fast flow
Strategies for structuring a monolith that improve team autonomy
Design techniques for reducing build-time coupling and accelerating

In chapter 5, I described the architectural requirements for fast flow in terms
of five quality attributes: modifiability, evolvability, testability, deployability
and observability. Throughout that chapter, I discussed the role that an
application’s component architecture - and its build architecture - plays in
satisfying these quality attributes. In particular, I showed how a multi-
component architecture is often necessary to meet these requirements in
large applications developed by large organizations.
However, despite its limitations, a single component architecture, such as the
monolithic architecture, is still a good choice for many applications. After
all, it avoids much of the complexity inherent in distributed, multi-
component architectures, such as the microservice architecture. In this
chapter, I describe the monolithic architecture and how well it satisfies the
architectural requirements for fast flow. You will also learn how to improve a
monolith’s ability to support fast flow by carefully structuring the parts of
the architecture that the monolithic architecture style leaves unspecified. In
particular, I show how to structure a monolith to improve team autonomy
and reduce build-time coupling.
Let’s start by defining the monolithic architecture and reviewing its benefits
and drawbacks.

6.1 About the monolithic architecture
The monolithic architecture is a simple architectural style that structures an
application as a single component. As shown in figure 6.1, the code resides
in single code repository and it is built and deployed by a single deployment
pipeline. The application often has a single relational database.
Figure 6.1 The monolithic architecture is an architectural style that structures the application as
a single component in a single code repository that’s built and deployed by a single deployment
pipeline
Because of its simplicity, the monolithic architecture is a suitable
architecture for many applications - especially early in its lifetime. And as I
describe in sections Section 6.2 and Section 6.3, there are design techniques
that can improve a monolith’s ability to support fast flow.
In this section, I describe the benefits of the monolithic architecture and its
drawbacks - including why its ability to support fast flow declines as the
application and its organization grow. Let’s start by looking at the monolithic
architecture’s benefits.
6.1.1 Benefits of the monolithic architecture
The monolithic architecture has several benefits:
Simplicity of a single code base
System operations are easier to implement, understand and maintain
because they are local to a single component
Simplified deployment since there’s only a single component

---

## Page 191

Simplified troubleshooting of runtime issues
Let’s look at each one in turn.
SIMPLICITY OF A SINGLE CODE BASE
One important benefit of the monolithic architecture is that developing a
monolith, which has a single code base, is generally much simpler than a
distributed application. Most notably,
Easy to make radical changes to the application - You can change the
code and the database schema, build, and deploy.
Simple to develop - IDEs and other developer tools are focused on
building a single application.
Compile-time enforcement of APIs - when using a compiled language,
such as Java or Rust, the compiler verifies that interfaces are used
correctly instead of needing to use more complex techniques, such as
contract testing, which I discuss in chapters 15 and 16.
SYSTEM OPERATIONS ARE EASIER TO IMPLEMENT,
UNDERSTAND, AND MAINTAIN
Another important benefit of a monolithic architecture is that system
operations are, by definition, local to a single component. Not only do most
operations not incur the overhead of communication over a network, but
they can be implemented as a single transaction. As a result, they are
typically easier to implement, understand and troubleshoot.
As I describe in chapter 10, such operations are Atomic Consistent Isolated
Durable (ACID). Operations have a consistent view - the 'C' in ACID - of the
data. The database ensures that concurrently executing operations are
isolated from one another - the 'I' in ACID. In contrast, in a microservice
architecture, operations that span services are implemented as a series of
transactions in each of the services. Consequently, they are not automatically
isolated from one another, and can access inconsistent data.

---

## Page 192

There are, of course, exceptions to the one transaction per system operation
rule. Bulk operations, such as batch jobs, are often intentionally split into
multiple transactions in order to improve error handling, performance and
scalability. Additionally, as I describe below in section Section 6.2, it’s
sometimes beneficial to split the implementation of an operation into
multiple parts, each of which is a separate transaction. For example, the
createOrder() operation could be split into a synchronous part that creates
an Order and sends an event to a message broker, and one or more event
handlers that complete the creation of the Order by, for example, sending a
confirmation email.
Most operations can, however, be implemented as single transactions.
Moreover, it’s important to note that this is a decision that’s entirely up to the
developer - they are not forced to do this because the operation spans
multiple components.
SIMPLIFIED DEPLOYMENT
Another benefit of the monolithic application is that because it consists of a
single component, some aspects of deployment are usually straightforward.
Unlike with a multi-component architecture, a monolith has a single
technology stack. As a result, deployment is relatively simple - usually a set
of load balanced component instances that are deployed using one of the
technologies described in chapters 18 and {chapter-kubernetes}.
However, even though there’s a single component, the monolithic
architecture supports more sophisticated deployment architectures that can,
for example, allow different subdomains to be scaled separately. For
example, let’s imagine that the FTGO application’s Restaurant Management
subdomain has a large in-memory database of restaurants in order to support
fast queries. Deploying a single cluster of component instances would likely
be inefficient, as each instance would require a significant amount of
memory—even if most subdomains don’t need it. A more efficient approach
is a multi-cluster deployment, which is shown in figure 6.2, where each
cluster handles different requests and runs on appropriately sized hardware.

SIMPLIFIED TROUBLESHOOTING
Another benefit of the monolithic architecture is simplified troubleshooting.
Since operations are typically local to a single component instance, all
relevant log entries for a request are in a single log file. As a result,
troubleshooting runtime problems is more straightforward than in a
distributed architecture where the relevant log entries are often spread across
multiple files.
An exception to this occurs when an operation is split across multiple
component instances. In such cases, the application must implement some of
the observability patterns described in chapter 17, such as log aggregation
and distributed tracing, to enable effective troubleshooting.
As you have seen, the monolithic architecture has numerous benefits. The
problem is that an application and its organization can outgrow the
monolithic architecture. Eventually, the monolithic architecture becomes an
obstacle to fast flow. To understand why, let’s look at how well a monolith
satisfies the five quality attributes for fast flow.
6.1.2 The monolithic architecture and the quality attributes for
fast flow
In chapter 5, I described five architectural quality attributes for fast flow:
modifiability, evolvability, testability, deployability and observability. A
small monolith developed by a small organization can often satisfy these
attributes quite well. The challenge, however, is with large monoliths
developed by large organizations. In those scenarios, the monolithic
architecture can be an obstacle to fast flow. To understand why, let’s examine
how well the monolithic architecture supports each of these quality
attributes, starting with modifiability.
MODIFIABILITY: REQUIRES DISCIPLINED DESIGN
Modifiability is an essential quality attribute for fast flow. As described in
chapter 5, modifiability is achieved by having an architecture whose domain
and component dimensions are loosely design-time coupled and cohesive.

---

## Page 195

Fortunately, there’s no reason why a monolith’s domain view cannot be
loosely design-time coupled - it’s simply a matter of applying good design
principles. Also, since a monolith consists of a single component, there’s no
possibility of design-time coupling within the component view.
One challenge with ensuring that a monolithic architecture is loosely design-
time coupled, however, is maintaining the discipline to continually practice
good design. The monolithic architecture is inherently very forgiving. You
can, for example, add a new class almost anywhere in the code base and it
will work. Schedule pressure can encourage developers to take shortcuts that
create tight design-time coupling. Over time, these compromises can
accumulate as technical debt, increasing the risk that the application
devolves into a big ball of mud.
Later in section Section 6.2, I’ll describe how another challenge with the
monolithic architecture is that the patterns that we typically use to organize
the domain view, such as a layered architecture, don’t scale to support the
growing complexity. These patterns work well in a small monolith, but as it
grows, they fail to provide the loose design-time coupling and cohesion that
fast flow requires. In that section, I describe a more effective alternative: the
modular monolith.
EVOLVABILITY: DECLINES AS THE APPLICATION AND ITS
ORGANIZATION GROW
In order to support fast flow, a monolith must be highly evolvable. However,
as I described in chapter 5, when a large application is developed by a large
organization, high evolvability is best achieved through a multi-component
architecture. Since a monolithic architecture consists of a single component,
its evolvability declines as both the application and the organization grow.
As figure 6.3 shows, upgrading the technology stack of a large monolith
often requires coordination across all teams, which is time-consuming and
slows down development. Widespread changes to a large code base are
major undertakings that cannot easily be done incrementally. Even critical
upgrades—such as those required to fix security vulnerabilities—can take a
long time, increasing operational risk.

Testability is another quality attribute that is essential for fast flow. A small
monolith developed by a small team can be highly testable. But, as I

Let’s examine in more detail how a monolith struggles to satisfy those three
dimensions of deployability as it and its organization grow. The first
problem is that the speed of deployments declines. Deployments are often
fast for a small monolith developed by a small organization. However, as the
monolith and the organization grow, the deployment step of its deployment
pipeline increasingly becomes a bottleneck. While build acceleration
technologies, such as parallelization and clustering, can accelerate the tests,
deployments must be done sequentially. Larger applications also take longer
to deploy due to increased startup times. Some canary release strategies even
require the application to be started twice—once for the canary and again for
the full rollout—further slowing the process.
To make matters worse, slow deployments combined with high test
execution time increase the deployment pipeline’s utilization, reducing its
ability to handle the high volume of changes produced by a large
organization. As a result, the deployment pipeline becomes overloaded:
changes are either queued—leading to longer lead times—or batched
together, making production issues harder to troubleshoot.

The second problem is that deployment reliability declines as the monolith
grows. Because every deployment updates the entire application—even if
the change itself is small—the risk of unintended consequences increases
significantly. In a large, complex monolith, a seemingly minor change can
inadvertently impact unrelated parts of the system due to hidden
dependencies, shared configuration, or gaps in test coverage. As a result, the
frequency of deployment-related failures tends to rise.
The third problem is that the ability of teams to deploy their changes
autonomously declines as the organization grows. Since a monolith consists
of a single component, because all teams contribute to the same code base.
They cannot deploy their changes independently. One team could, for
example, break the build, which prevents other teams from deploying their
changes. The problems caused by the lack of independence increase as the
organizations grow.
The fourth and final problem is that time to verify a deployment increases as
the monolith grows. For example, canary releases observe the behavior of
the new version of the application in production before rolling it out to all
users. Essentially, the application is tested using production traffic. The
challenge with a complex monolith is that since it implements so much
functionality, it can take longer to be confident that the new version works. It
might even be impractical to test the entire application as part of a canary
release. You might, for example, introduce a bug that won’t be detected until
a batch job runs days later.
OBSERVABILITY: DECLINES AS THE APPLICATION GROWS
Observability is the fifth and final quality attribute essential for achieving
fast flow. A monolithic application can be observable by simply
implementing the observability patterns described in chapter 17. However,
the monolithic architecture has a significant drawback: since the application
consists of a single component, telemetry such as resource utilization and
performance metrics are aggregated across all subdomains. Consequently,
visibility into the behavior of individual subdomains is limited. As the
monolith grows larger and more complex, understanding and diagnosing
issues within specific subdomains becomes increasingly difficult.

6.1.3 Other drawbacks of monolithic architecture
Declining evolvability, testability, deployability, and observability are not the
only drawbacks of the monolithic architecture. Two other significant
drawbacks are:
Inability to support multiple technology stacks
Inability to segregate subdomains by their characteristics
Let’s look at each of these drawbacks in turn.
INABILITY TO SUPPORT MULTIPLE TECHNOLOGY STACKS
Earlier in section Section 6.1.2, I described how the evolvability of a
monolithic architecture declines as the application and its organization grow.
A related drawback is that since a monolith consists of a single technology
stack, all code must use the same language runtime and supporting libraries.
For example, a Java monolith must use JVM-compatible languages. This
limitation prevents teams from using the best tool for each subdomain even
though other languages or runtimes might offer better performance, a more
appropriate abstraction, or access to a specialized ecosystem. But in a
monolith, mixing languages or runtimes is not possible.
INABILITY TO SEGREGATE SUBDOMAINS BY THEIR
CHARACTERISTICS
Another drawback of the monolithic architecture is its inability to segregate
subdomains based on their characteristics. As I describe in chapter 20,
subdomains differ in characteristics such as business criticality, and security.
In a microservice architecture, it is often advantageous to isolate subdomains
with different characteristics into separate services. For example,
subdomains that handle sensitive data or implement business-critical
functionality can be deployed as separate services that are isolated from the
rest of the application. A monolithic architecture, however, lacks this
flexibility: all subdomains are packaged and deployed together, without the
structural separation that services provide.

6.1.4 The monolithic architecture pattern language
As we’ve seen, the monolithic architecture can become a significant obstacle
to fast flow as both the application and its organization grow. Some
limitations — such as the use of a single technology stack — are inherent
and unavoidable. However, there are ways to mitigate some of the other

6.2 Modularizing the monolith
Despite having the reputation of being difficult to maintain, a monolithic

achieving this is the way that the domain dimension of a monolith is
traditionally structured: a technology-oriented architectural style. This style

cohesion. Worse still, a technically oriented architecture often increases
build-time coupling, which further reduces testability and deployability.

6.3.1 About physical design principles
Domain modules are part of the application’s domain architecture, but they
must be organized into build projects, which belong to the component
architecture. A common and seemingly straightforward approach is to map
each domain module to a single build project. However, this simple one-to-
one mapping often results in high build-time coupling—both between
modules and within a module—which can slow down testing and
deployment.
To understand why, consider a simple example with two domain modules:
Consumer Domain Module and Order Domain Module, where the later uses
the former. When mapped one-to-one, each one becomes a Gradle project -
consumer-domain and order-domain - then order-domain depends upon
consumer-domain. As a result, any change to consumer-domain will trigger
the recompilation and retesting of order-domain. In a real-world application,
such a change could cascade through multiple transitive dependencies,
significantly increasing build time. This slows down both the edit-test-
analyze loop for developers and the deployment pipeline.
A better approach is to organize domain modules into build projects using
'physical design principles'. Whereas logical design is concerned with
concepts such as naming and design-time coupling, physical design is
concerned with structuring the build architecture in way that minimizes
build-time coupling. It applies two key ideas from chapter 4 that minimize
build-time coupling both between and within projects. The first idea is that a
build project should only depend upon stable projects. The second idea is
that build projects should consist of elements that need to be tested together.

PHYSICAL DESIGN PRINCIPLES
One of my favorite books from the 1990s is Large Scale C++ Design by
John Lakos (Addison-Wesley, 1996). While much of the book is C++-
specific (not my most favorite language), it makes an important
contribution: physical vs. logical software design. Logical software design
is the main focus of software design techniques, such as object-oriented
design and Domain-Driven Design: naming, assignment of responsibilities
to classes and packages, class hierarchies, encapsulation, loose (design-

software to be understandable and maintainable.
Lakos argues, however, that for large-scale software, logical design is

6.4.1 A modular monolith is still a monolith
Despite its advantages, a modular monolith remains a monolith: a single

coarse-grained. Eventually, your application and organization may outgrow
the monolithic architecture. When that happens, you will need to use the
microservice architecture in order to achieve fast flow.

microservice architecture. There’s a common belief that when the time is

around consistency, concurrency, error handling, and rollback—complexity
that doesn’t exist within a monolith.

One downside of contract tests is the overhead of maintaining them. But this
effort is worth it because contract testing makes independently deployability
possible. I once encountered an organization that abandoned contract tests
because they were "too much work" and fell back on application-level end-
to-end tests. But I strongly suspect that the root cause was their excessively
fine-grained and tightly design-time coupled architecture. The fine-grained
architecture increased the number of contracts that needed to be maintained,
while the tight design-time coupling meant the contracts were constantly
changing. This example highlights the importance of the microservice
architecture’s other essential characteristic: loose coupling, which I describe
shortly.
BENEFITS OF INDEPENDENTLY DEPLOYABLE SERVICES
Independent deployability has two main benefits:
It accelerates, simplifies and increases the reliability of the deployment
pipeline
It reduces the need for cross-team coordination
Let’s look at each of these benefits in turn.
The first benefit of independent deployability is that it accelerates, simplifies
and increases the reliability of the deployment pipeline by eliminating slow,
complicated, and unreliable end-to-end tests. Imagine writing an end-to-end
test for a service that retrieves orders from the Order Service. Not only
would it need to deploy the Order Service, but it would have to deploy and
initialize all of its dependencies. It would, for example, need to create a

---

## Page 256

Consumer in the Consumer Service before it could create an Order in the
Order Service. Such a test is likely to be slow and complicated.
Worse, it might fail randomly due to the complexity of the application.
Teams would need to investigate the failure to determine whether it can
safely be ignored. This introduces a frustrating manual step into what should
be a fully automated deployment pipeline. Application-level end-to-end tests
that are run before releasing the entire application are even worse.
In contrast, a service-level test that tested the service in isolation would only
need to deploy the Order Service, create one or more orders, and then
query the Order Service. Any dependencies of the Order Service that are
needed to create orders, such as the Consumer Service, would be simulated
using test doubles. The test would be relatively simple, fast and reliable.
The second benefit of independent deployability is that it reduces the need
for cross-team coordination. When services are tested together, a failure
caused by one team’s change can block another team’s release. Teams must
also collaborate to debug failures of application-level end-to-end tests. This
tight coupling slows down the teams and limits each team’s ability to work
independently.
Together, these benefits make independent deployability a major enabler of
fast, sustainable flow. It allows teams to move quickly and safely, releasing
changes without waiting on or being blocked by other parts of the
organization. Without it, the benefits of microservices are much more
difficult to achieve. Yet, as I describe below, a common antipattern of
microservices adoption is an organization insisting on testing the
application’s services together.
As you can see, independent deployability is an important characteristic of
the microservice architecture. Without it, you have a distributed monolith.
Yet sadly, I’ve encountered numerous organizations that insist on testing the
application’s services together. Let’s look at one of the most common
obstacles to independently deployable services: application-level user
acceptance tests.

7.3.2 Essential characteristic #2: Services are loosely coupled
The second essential characteristic of the microservice architecture is that
the services are loosely coupled. As I described chapter 4, there are four
flavors of coupling:
Design-time coupling - the likelihood that a change to one software
element will require a change to the other element.
Build-time coupling - when changing one element requires another
element to be recompiled and retested.
Runtime coupling - when one element’s failure causes another element
to fail.
Infrastructure coupling - when two elements share infrastructure, such
as a database, and one element’s behavior affects the other element
Let’s look at each of these flavors of coupling in the context of a
microservice architecture, starting with design-time coupling.
SERVICES MUST BE LOOSELY DESIGN-TIME COUPLED
Design-time coupling between services is a serious concern because of the
high cost of lock step changes - changes that must be coordinated across
multiple services, or worse, across multiple teams. Tight design-time
coupling significantly reduces an application’s modifiability is a major
obstacle to fast flow.
For example, let’s imagine that the Consumer Management team needs to
make a breaking change to the Consumer Service’s API. They can’t
simply make the change and deploy the new version since that
would break its clients - such as the `Order Service - and cause a
production outage. Instead, the team must:
1. Add a new version of the Consumer Service's API alongside the
current version.
2. Ask the teams that own the client services - including the Order
Service's team - to migrate to the new version.

---

## Page 258

3. Once all clients have migrated, the Consumer Management team can
delete the old version.
As you can see, breaking API changes are a major burden - a topic I explore
further in chapter 3. And, as I describe below, one of the inherent risks of the
microservice architecture. That’s why loose design-time coupling is
essential.
SERVICES SHOULD BE LOOSELY RUNTIME COUPLED
Runtime coupling can also be a significant issue in a microservice
architecture. Tight runtime coupling doesn’t prevent an architecture from
enabling fast flow. But as chapter 4 describes, it increases latency and
reduces availability. You must carefully design your service architecture to
prevent runtime coupling from becoming a problem. When defining a
service architecture, you might find that to avoid runtime coupling, you will
need to colocate multiple subdomains in a single service. Chapter 20
discusses this in more detail.
BUILD-TIME COUPLING IS USUALLY NOT AN ISSUE
Build-time coupling is usually not an issue in a microservice architecture.
Since services are independently deployable components that are built and
tested separately, there is no build-time coupling between services. There
might be build-time coupling within a service, but the codebase is typically
small, and the deployment pipeline remains fast. But if necessary, you can
often accelerate the pipeline by applying physical design principles I
described in chapter 6 to organize the codebase.
INFRASTRUCTURE COUPLING IS MOSTLY A MATTER OF CHOICE
The degree of infrastructure coupling depends on the chosen deployment
architecture. For example, an organization can choose whether to use a
shared database server or a database service for each service. This decision
involves trade-offs between the benefits of shared infrastructure and the
drawbacks of infrastructure coupling.

7.4.1 Enables fast, sustainable flow
The biggest benefit of the microservice architecture is that it enables fast

evolvability, testability, deployability and observability. Let’s look at how
the microservice architecture satisfies each of these attributes.
HIGH MODIFIABILITY DUE TO LOOSE DESIGN-TIME COUPLING
One key reason why the microservice architecture supports fast flow is that
it has high modifiability. By definition, the services are loosely design-time
coupled. A change to one service rarely requires a change to another service.
As a result, the teams that own the services work independently of one
another with a high degree of autonomy.
Of course, designing services that are loosely design-time coupled requires

architecture, such as the microservice architecture, is more evolvable than a
monolithic architecture. The application’s technology stack can evolve
incrementally, one component - or in this case, one service - at a time.
Upgrades are broken down into smaller, per-team efforts that are easier to
schedule and prioritize based on risk and need. Also, as I describe below in

technology stack that is best suited for what it needs to.
HIGH TESTABILITY DUE TO SMALLER, INDEPENDENTLY

The third reason the microservice architecture supports fast flow is that it has
high testability. Since the application consists of services that are small and
testable in isolation without complex and slow end-to-end tests, it’s almost

execute locally. Each service also has its own deployment pipeline, which
typically handles a relatively low volume of changes. As a result, testing is
unlikely to become a bottleneck.
Moreover, the microservice architecture allows subdomains that are
inherently difficult to test — such as those requiring manual testing, slow-
running test suites, or specialized environments — to be isolated in their
own services. This ensures that the testability of other services is not
compromised.
HIGH DEPLOYABILITY
The fourth reason the microservice architecture supports fast flow is that it
has high deployability. Each service is relatively small and independently

HIGH OBSERVABILITY
The fifth and final reason the microservice architecture supports fast flow is

subdomain. This can make it easier to understand resource utilization and
diagnose problems.
As you can see, the microservice architecture enables a large application -

sustainable flow. While enabling fast flow is the primary benefit, it’s not the

7.4.2 Supports multiple technology stacks
Not only does the multi-component nature of the microservice architecture
improve evolvability, but it also enables each service to use a technology

the right job strategy. This does not mean, however, complete anarchy. To
properly benefit from the freedom to choose, an organization must still
carefully manage the set of approved technologies.
Another related benefit of the microservice architecture is that
experimenting with new technologies is easier and safer. For example, a
team can implement a service using a new technology stack and deploy it
into production. If the experiment succeeds, the organization has validated a
new stack for future use. Otherwise, if the experiment is unsuccessful, then
it’s not a disaster. If it fails, the impact is limited — valuable lessons are

7.4.3 Enables an organization to scale
Another important benefit of the microservice architecture is that it enables
an organization to scale effectively without sacrificing fast flow. Even a
well-structured monolith eventually becomes an obstacle to fast flow as the
organization grows. A single deployment pipeline cannot handle the volume
of commits and becomes a bottleneck, and tight design-time coupling makes
matters worse. Adding developers can actually slow things down due to the
increased overhead of coordinating the work of multiple teams. The book
Wiring the Winning Organization by Gene Kim and Dr. Steven J. Spear (IT
revolution press, 2023) describes how the pace of software delivery at
Amazon.com slowed to a crawl in 2002 despite their growth.
In contrast, the microservice architecture enables sustainable scale. New
teams can take ownership of new services with minimal coordination costs
— something that’s impossible to achieve in a tightly coupled system. Each
service has its own deployment pipeline, and so the build architecture scales
as the organization grows.
As a result, the organization can continue to deliver features rapidly, even as
it grows. Wiring the Winning Organization described how Amazon
overcame its software delivery problems by adopting a microservices-style
architecture - referred to in the book as modularization. By 2025, Amazon
was reportedly doing 130,000 deployments per day.

7.5.1 Distributed operations are complicated
One significant drawback of the microservice architecture is that designing
distributed operations is much more complicated than local operations. A
local system operation can execute with a transaction and simply call
methods on various classes - just like in a monolith. But as figure 7.8,
services in a microservice architecture are both process and transaction
boundaries. As a result, a distributed operation - one whose subdomains
reside in different services - will span those boundaries, which makes it far
more complicated to implement.

The risk of runtime coupling
The complications of eventual consistency

THE COST AND LATENCY OF NETWORK COMMUNICATION
One challenge with distributed operations is the significant cost and latency
of network communication. Compared to a method call - measured in
nanoseconds - communication over a network is staggeringly expensive,
typical in the milliseconds. What’s more, a network bandwidth is finite and
over-utilization will increase latency and reduce throughput.
As a result, it’s essential to consider the cost of distribution when designing a
microservice architecture. However, it’s important to remember that the
relative cost is not the same as actual impact. What matters is whether a
distributed operation meets its Service Level Objective (SLO). If its latency
is acceptable, then the cost of communication might not be an issue.
The real challenge arises when making an operation distributed would
prevent it from meetings its SLO. This scenario involves a potentially
difficult trade-off between performance and the benefits of distribution, such
as team autonomy. For example, imagine that Order Management and
Consumer Management subdomains were packaged as separate services. If
doing so causes createOrder() to exceed its latency SLO then you have
two options:
Accept the high latency and tolerate the impact on user experience
Combine the two subdomains into a single service so that it can meet its
SLO
The second option enables createOrder() to meet its SLO, but it comes at a
cost: it reduces the autonomy of the two teams that own those subdomains.
They must coordinate changes and deployments which can slow down
development.
Designing a microservice architecture means making these kinds of trade-
offs deliberately and with full awareness of their consequences.

RISK OF RUNTIME COUPLING
Another challenge with distributed operations is the risk of runtime
coupling. As I described chapter 4, runtime coupling occurs when one
service cannot respond to a request - the invocation of a system operation -
without a receiving a response from another service. This dependency
reduces availability and increases latency.
When designing a microservice architecture, it’s essential to assess each
distributed operation’s potential for runtime coupling. In some cases, you
can reduce or even eliminate an operation’s runtime coupling by carefully
defining service boundaries. For other operations, runtime coupling may be
unavoidable unless you modify the operation’s semantics to allow partial
outcomes or make the operation asynchronous. It’s important to remember
that it’s only an actual problem if it causes an operation to not satisfy its
SLOs.
It’s important to remember that runtime coupling is only a problem if it
prevents an operation from meeting its SLOs. If an operation is sufficiently
performant and available, then any runtime coupling might not be an issue.
But if it introduces significant latency or reduces availability, then you need
to consider architectural changes.

Another major challenge with distributed operations is transaction
management. System operations typically read and write persistent data, and
so need to be transactional. In a monolith, system operations are simply
executed within an ACID database transaction (e.g., using Spring’s
@Transactional). If the operation is successful, the transaction is
committed. Otherwise, it is rolled back.
But in a microservices architecture, transaction management is far more
challenging. To understand why, consider the following scenario:
Consumer Service - implements the local operation createConsumer()

which invokes Customer Service.reserveCredit() to reserve credit
for the customer
A local operation, such as createConsumer(), can execute within a standard
ACID transaction, just like in a monolithic application. The real challenge,
however, is with distributed system operations, such as createOrder(),
which span multiple services.
Implementing transaction management for distributed operations is
complicated because of the need loose design-time coupling - a defining
characteristic of the microservice architecture. Services must not access each
other’s databases directly. Instead, they collaborate via APIs. What’s more,
each service can have its own database server in order to improve scalability
and reduce infrastructure coupling.
The traditional solution to distributed transactions is the two-phase, commit
(2PC) protocol. However, as you will see in chapter 10, 2PC is not a good
choice for modern applications. It’s essentially a form of runtime coupling
that can lead to availability and latency problems. In addition, many modern
databases do not support 2PC.

to within a service and, implement distributed operations as a sequence of

eventual consistency.

7.5.2 Implementing changes that span services is complicated
One of the great strengths of the microservice architecture is that making a
change to one or more subdomains within that are in the same service is
faster and easier than making the equivalent change in a monolith. Since
each service is small and independently deployable, changes can be
developed, tested, and released without coordination with other teams.
Yet ironically, the inverse is also true: implementing changes that span
multiple subdomains in multiple services is more complicatedly complicated
than making the equivalent change in a monolith. In a monolith, a large
change can be made by a single commit or pull request. In contrast, in a
microservice architecture requires such changes to be carefully planned and
sequenced to avoid production outages.
Let’s consider a concrete example. Suppose the Order Management team is
developing an enhancement to the Order Service that requires the
Consumer Management team to change the Consumer Service's
reserveCredit() operation to return the customer’s available credit. This is
an example of a backwards compatible change to the Consumer Service's
API. Old clients can continue to use the updated API by simply ignoring the
new availableCredit property in the response. But from a deployment
perspective, there’s an ordering dependency. The updated Consumer
Service must be deployed before the Order Service can use it.

Now let’s imagine that instead of a backwards compatible change, the
enhancement requires a breaking change to the Consumer Service's API.
The Consumer Management team cannot simply deploy the breaking change
since that would cause a production outage. Instead, a sequence of non-
breaking changes must be made:
1. The Consumer Management must add a new version of the Consumer
Service API alongside the old version. One common approach is to
prefix the URL paths with the version, e.g. /v1/…​ vs. /v2/…​.
2. The client teams, including the Order Management team, must migrate
their services to the new API version - perhaps after nudging from by
the Consumer Management team
3. Once all clients had migrated to the new version of the Consumer
Service API, the Consumer Management team can then delete the old
API version.
As you can see, rolling out a breaking change to a service’s API is much
more complicated and time-consuming than the equivalent change in a
monolith.
These kinds of cross-service changes highlight one of the core trade-offs of
microservices: you gain autonomy and speed for local changes, but you pay
for it when implementing distributed changes across subdomains. A key
design goal is to define service boundaries so that most changes are local to
a single service. Unfortunately, achieving this is another challenge of the
microservice architecture.
7.5.3 Designing a good microservice architecture is challenging
Software design is largely about naming and grouping things, and applying
the principles of loose design-time coupling and high cohesion, which I
described in chapter 4. While it’s inherently challenging, it’s especially
challenging to design a good microservice architecture that enables fast flow
while being performant and highly available. That’s because services are
process and transaction boundaries. As a result, the decision to place two
subdomains that are invoked by the same system operation in separate

---

## Page 276

services is far more consequential than, say, putting two classes in different
packages.
It makes the microservice architecture very unforgiving of design mistakes.
It’s far too easy to create a distributed monolith: services that are (a) tightly
design-time coupled so changes ripple across services, and (b) tightly
runtime coupled so a failure in one service cascades into another. Careful,
rigorous design is required to maximize the benefits of the microservice
architecture and minimize its drawbacks.
As you will see in chapter 20, designing a microservice architecture means
navigating a complex design space filled with design alternatives. For
example, let’s imagine that you are implementing a new feature that
introduces a new subdomain and a new operation that spans both the new
subdomain and one or more of the existing subdomains. A key design issue
is: which service should contain the new subdomain? If the architecture
consists of N services, then there are N+1 options - an existing one or a new
service.
And for each one of those possible service designs, there are likely multiple
ways to implement the system operation. As you will learn in chapters 10
and 11, there are often multiple ways to implement a system operation -
multiple candidate patterns and multiple ways to implement each pattern -
each of which involves different trade-offs. What’s more, one design
candidate might simplify the implementation of the operation at the expense
of a less than ideal service design, or vice versa. These trade-offs often
conflict, and evaluating the design space requires thoughtfulness and
discipline.
7.5.4 Operational challenges
Not only is designing a microservice architecture challenging, but so is
deploying and operating it. A monolith’s deployment architecture consists of
multiple instances of a single application. In contrast, the runtime
architecture of a microservice architecture consists of multiple instances of
multiple services - each potentially using different versions of different
technology stacks. Moreover, microservice-based systems can exhibit
emergent behavior—unexpected system-level issues that arise from

---

## Page 277

interactions between independently developed and deployed services. These
issues are often difficult to understand and troubleshoot.
Fortunately, as you will see in chapters 18 and {chapter-kubernetes}, there
are many tools and techniques that can help. Container technologies allow a
production environment to treat a service as a black box that hides its
technology stack. Sophisticated orchestration platforms, such as Kubernetes,
simplify the deployment and operation of services. And, modern
observability tools make it easier to observe and troubleshoot a
microservices-based application. Nevertheless, these tools can only help so
much, and so some operational challenges still remain.
Despite these operational challenges, many organizations still pursue the
microservice architecture for its long-term benefits. But deciding when and
how to adopt it — especially in an existing system — is yet another
challenge.

TESTS
Application-level user acceptance tests are typically written from the

7.6.1 Patterns and pattern languages
A pattern is a reusable solution to a problem that occurs in a particular
context. The concept originated in real-world architecture, but it has since
become foundational in software architecture and design. The concept of a
pattern was created by Christopher Alexander, a real-world architect, who
also introduced the concept of a pattern language: a collection of related
patterns that solve problems within a particular domain. His book A Pattern
Language: Towns, Buildings, Construction (Oxford University Press, 1977)
describes a pattern language for architecture that consists of 253 patterns.
The patterns range from solutions to high-level problems, such as where to
locate a city ("Access to water"), to low-level problems, such as how to
design a room ("Light on two sides of every room"). Each pattern is a
proven solution that arranges physical elements - ranging in scope from
cities to windows.
Alexander’s ideas inspired the software community to adopt the concept of
patterns and pattern languages. The landmark book Design Patterns:
Elements of Reusable Object-Oriented Software (Addison-Wesley
Professional, 1994), by Erich Gamma, Richard Helm, Ralph Johnson, and
John Vlissides is a collection of object-oriented design patterns and helped
popularize patterns among software developers. Since the mid-1990s,
software developers have documented numerous software patterns. A
software pattern solves a software architecture or design problem by
defining a set of collaborating software elements.
Let’s imagine, for example, that you’re building a banking application that
must support a variety of overdraft policies. Each policy defines different
limits and the fees. This problem can be solved using the Strategy pattern,
which is a well-known pattern from the classic Design Patterns book. The
Strategy pattern consists of three parts:
A strategy interface called Overdraft that encapsulates the overdraft
algorithm
One or more concrete strategy classes, one for each particular context
The Account class that uses the algorithm

---

## Page 283

Because the Strategy pattern is an object-oriented design pattern, the
elements of the solution are classes. Later in this section, I describe higher-
level design patterns, where the solution consists of collaborating services.
Patterns are valuable not only because they provide reusable solutions but
also because a pattern must describe the context within which it applies. The
idea that a solution is specific to a particular context and might not work
well in other contexts is an improvement over how technology used to
typically be discussed. This is especially important in software architecture,
where, for example, solutions work for Netflix might not be the best
approach for smaller-scale applications.
But the value of a pattern goes far beyond requiring you to consider the
context of a problem. A well-defined pattern describes other critical yet
frequently overlooked aspects of a solution. A typical pattern structure
includes three especially valuable sections:
Forces
Resulting context
Related patterns
Let’s look at each of these, starting with forces.
FORCES: THE ISSUES THAT YOU MUST ADDRESS WHEN
SOLVING A PROBLEM
The forces section of a pattern describes the forces or concerns that you must
address when solving a problem in a given context. Forces can conflict, so it
might not be possible to solve all them. Which forces are more important
depends on the context. You have to prioritize solving some forces over
others. For example, you often want code to be easy to understand and have
good performance. Code written in a reactive style has better performance
than synchronous code, yet is often more difficult to understand. In some
contexts, performance is more important than understandability; in others,
the opposite is true. By explicitly listing the forces, you make clear which
issues need to be solved.

PATTERN
The resulting context section of a pattern describes the consequences of
applying the pattern. It consists of three parts:
Benefits — The benefits of the pattern including the forces that have
been resolved
Drawbacks — The drawbacks of the pattern including the unresolved
forces
Issues — The new problems that have been introduced by applying the
pattern
The resulting context provides a more complete and less biased view of the
solution, which enables better design decisions.
RELATED PATTERNS: THE FIVE DIFFERENT TYPES OF
RELATIONSHIPS
The related patterns section of a pattern describes the relationship between
the pattern and other patterns. There are five types of relationships between
patterns:
Predecessor — A predecessor pattern is a pattern that motivates the
need for this pattern. For example, the Microservice architecture pattern
is the predecessor to the rest of the patterns in the pattern language,
except the monolithic architecture pattern.
Successor — A pattern that solves an issue that has been introduced by
this pattern. For example, if you apply the Microservice architecture
pattern, you must then apply numerous successor patterns, including
service discovery patterns and the Circuit breaker pattern.
Alternative — A pattern that provides an alternative solution to this
pattern. For example, the Monolithic architecture pattern and the
Microservice architecture pattern are alternative ways of architecting an
application. You pick one or the other.

## Page 285

Generalization — A pattern that is a general solution to a problem. For
example, in chapter 12 you’ll learn about the different implementations
of the Single service per host pattern.
Specialization — A specialized form of a particular pattern. For
example, in chapter 12 you’ll learn that the Deploy a service as a
container pattern is a specialization of Single service per host.
In addition, you can organize patterns that tackle issues in a particular
problem area into groups. The explicit description of related patterns
provides valuable guidance on how to effectively solve a particular problem.

ONE-TO-ONE INTERACTION STYLES
There are three types of one-to-one interactions:
Request/response — A service client makes a request to a service and

timely fashion. It might even block while waiting. This interaction style
tends to create tighter runtime coupling between services.

waiting because the service might not send the response for a long time.
One-way notifications — A service client sends a request to a service,
but no reply is expected or sent.
It’s important to remember that the synchronous request/response interaction
style is mostly orthogonal to IPC technologies. A service can, for example,
interact with another service using request/response style interaction with

message broker, the client service might be blocked waiting for a response.
It doesn’t necessarily mean they’re loosely coupled. That’s something I
revisit later in this chapter when discussing the impact of inter-service
communication on availability.

Publish/subscribe — A client publishes a notification message, which is

Publish/async responses — A client publishes a request message and

8.1.2 Defining APIs in a microservice architecture
APIs, or interfaces, are central to software development. An application

8.1.3 APIs have Service Level Objectives (SLOs)
APIs are not just functional contracts; they also represent operational
promises. When a client calls a service, it expects not only correct behavior
but also a certain level of reliability and responsiveness. A service API must,
therefore, satisfy various non-functional requirements, also known as
software quality attributes, such as availability, performance, scalability,
security, and reliability. To make these requirements concrete and verifiable,
teams define metrics and targets, respectively known as service level
indicators (SLIs), and service level objectives that quantify the desired
service behavior.

Service Level Indicators (SLIs) - quantitative metrics that capture

level-objectives/) - target values for those SLIs that express the desired
level of reliability or performance.
Service Level Agreements (SLAs) - SLOs with contractual
commitments, often with penalties or credits for violations.
An SLO defines the target for an SLI, while an SLA formalizes that target
for clients.

8.1.4 API Security
Security is a critical aspect of any API design. Even a perfectly designed API
is dangerous if it allows unauthorized access or leaks sensitive data. Security
is a broad topic, but three key concerns are
Secure communication - protecting data in transit from interception or
tampering
Authentication - verifying the identity of the caller
Authorization - controlling what authenticated identities are allowed to
do
Let’s look at each one of these in turn.

SECURE COMMUNICATION
Secure communication ensures that data exchanged between clients and
services cannot be intercepted or altered. In practice, this means that all API
traffic should use Transport Layer Security (TLS). TLS encrypts data in
transit, prevents man-in-the-middle attacks, and verifies the authenticity of
the service endpoint.
Establishing secure communication is the foundation of API security:
without it, even strong authentication and authorization mechanisms can be
bypassed by network-level attacks.
AUTHENTICATION
Authentication verifies who is making a request. APIs commonly use access
tokens — such as OAuth 2.0 bearer tokens or JSON Web Tokens (JWTs) —
to authenticate clients and securely convey identity claims. Depending on
the system, authentication may involve end-user credentials, client
certificates, or federated identity providers that issue tokens trusted across
multiple services.
AUTHORIZATION
Authorization determines what an authenticated identity is permitted to do. It
defines the actions and resources accessible to users or services once their
identity is established. Authorization policies are often enforced using role-
based or attribute-based access control models. For infrastructure services,
similar principles apply. For example, Kafka Access Control Lists (ACLs)
can restrict which producers and consumers are allowed to publish or
subscribe to specific topics.
8.1.5 Evolving APIs
APIs invariably change over time as new features are added, existing
features are modified, and sometimes old features are removed. In a
monolithic application, it’s relatively straightforward to change an API and
update all the callers. If you’re using a statically typed language, the

compiler helps by giving a list of compilation errors. The only challenge
may be the scope of the change - it might take a long time to change a
widely used API.
In a microservices-based application, changing a service’s API is far more
difficult. A service’s clients are other services that are developed and
deployed independently, usually by other teams. In some cases, the clients
may even be other applications owned by other organizations. You usually
can’t force all clients to upgrade in lockstep with the service. Also, because
modern applications are rarely down for maintenance, you will typically
perform a rolling upgrade of your service, meaning both the old and new
versions of a service will be running simultaneously. What’s more, as I
describe in section Section 8.3.4, in an asynchronous messaging architecture,
a new receiver must be able to handle messages sent by an old sender.
It’s important to have a strategy for dealing with these challenges. How you
handle a change to an API depends on the nature of that change. There are
also important differences between evolving a synchronous RPI (see section

8.1.6 Message payload formats
The essence of IPC is the exchange of messages. A message carries data -
also known as its payload - and so an important design decision is the format
of that data. The choice of message payload format can impact the efficiency
of IPC, the usability of the API, and its evolvability. If you’re using a
messaging system or protocols such as HTTP, you get to pick your message
payload format. Some IPC mechanisms — such as gRPC, which you’ll learn
about shortly — might dictate the message payload format. In either case,
it’s essential to use a cross-language message payload format. Even if you’re
writing your microservices in a single language today, it’s likely that you’ll
use other languages in the future. You shouldn’t, for example, use Java
serialization.
There are two main categories of message payload formats: text and binary.
Let’s look at each one in more detail.
TEXT-BASED MESSAGE PAYLOAD FORMATS
The first category is text-based formats such as JSON and XML. An
advantage of these widely used formats is that they are human-readable.
They are also self-describing, since a JSON message is a collection of
named properties, and an XML message is effectively a collection of named
elements and values. This format enables a consumer of a message to pick
out the values of interest and ignore the rest. Consequently, many changes to
the message schema can easily be backward-compatible.
The structure of XML documents is specified by an XML schema
(https://www.w3.org/XML/Schema). Over time, the developer community
has come to realize that JSON also needs a similar mechanism. One popular
option is to use the JSON Schema standard (http://json-schema.org). A JSON
schema defines the names and types of a message’s properties and whether
they’re optional or required. As well as being useful documentation, these
schemas can be used by an application to validate incoming and outgoing
messages.

---

## Page 311

One downside of using a text-based message format is that the messages
tend to be verbose, especially XML. Every message has the overhead of
containing the attribute names in addition to their values. Another drawback
is the overhead of parsing text, especially when messages are large.
Consequently, if efficiency and performance are important, you may want to
consider using a binary format.
BINARY MESSAGE PAYLOAD FORMATS
The second category of message formats is binary formats such as Protocol
Buffers (https://developers.google.com/protocol-buffers/docs/overview) and
Avro (https://avro.apache.org). These formats are designed to be compact
and efficient, which makes them a good choice for high-throughput systems.
The downside is that using them is more complicated because of the
additional tooling that’s required.
Both formats define a typed IDL for defining the structure of your messages.
An IDL compiler then generates the message classes along with code that
serializes and deserializes the messages. You’re forced to take an API-first
approach to service design! Moreover, if you write your client in a statically
typed language, the compiler checks that it uses the API correctly.
One difference between these two binary formats is that Protocol Buffers
uses tagged fields, whereas an Avro consumer needs to know the schema in
order to interpret messages. Each field in a Protocol Buffers serialized
message is identified by a numeric tag, allowing the consumer to parse
messages even if its local schema is missing some fields or has been updated
—unknown tags can simply be skipped. In contrast, Avro encodes data
without field identifiers, relying entirely on the writer’s schema to define the
order and types of fields. This design produces more compact messages but
requires the consumer to have access to the correct schema before it can
deserialize a message.
To make this practical in a distributed system where producers and
consumers evolve independently, Avro is typically used together with a
schema registry, such as the Confluent Schema Registry
(https://github.com/confluentinc/schema-registry). The Confluent Schema
Registry is a central service for storing and retrieving Avro schemas and

---

## Page 312

their versions, enforcing compatibility rules when new schemas are
registered. When a producer sends a message, it includes a small schema
identifier — assigned by the registry — in the message header. A consumer
receiving the message uses that identifier to look up the exact writer schema
from the registry and then performs schema resolution between the writer’s
and its own reader schema. For example, during schema resolution, the
consumer can fill in any missing fields with the default values defined in its
own reader schema.
The schema registry also has an API that enables build tools to enforce
compatibility. This mechanism allows Avro to maintain compact binary
encoding while supporting safe schema evolution and independent
deployment of producers and consumers.

8.2.1 Benefits and drawbacks of RPI
The Remote Procedure Invocation (RPI) pattern is widely used in distributed
systems and is often the default choice for service-to-service
communication. It enables a client to invoke an operation in a remote service
as if it were a local method call. This simplicity, however, comes with
important trade-offs that architects must understand when deciding between
synchronous and asynchronous communication styles.
BENEFITS OF THE RPI PATTERN
Simple and familiar - Most developers are already comfortable with the
request/reply model from programming languages and frameworks that
use function calls, HTTP APIs, or gRPC. This familiarity makes RPI
easy to understand, implement, and adopt.
Easy to implement the request/reply interaction style - RPI naturally
supports request/reply interactions, where a client sends a request and
waits for a response. This makes it straightforward to model query-style
operations and command-style requests that require immediate
confirmation.
Simpler system architecture - Since there is no intermediate message
broker, the system is easier to deploy, operate, and debug. Messages
travel directly from the client to the service, which can reduce latency
and operational complexity compared to a broker-based system.
DRAWBACKS OF THE RPI PATTERN
Runtime coupling between client and service - The client and service
must both be available and responsive at the same time for a request to
succeed. This tight runtime coupling means that failures, or high
latency in one service can quickly propagate to others. To mitigate these
risks, services must use resiliency patterns, which are described later in
section Section 8.2.4.
Limited interaction styles - RPI primarily supports the request/reply
style of interaction. It does not naturally support other interaction styles.

---

## Page 321

Supporting these patterns usually requires additional infrastructure,
such as a message broker or event streaming platform.
Requires service discovery - Clients must know the network location of
the service they want to call. A service discovery mechanism provides
this information, either by returning the network address of an available
service instance or by giving the client a load-balanced domain name or
IP address. I describe service discovery in more detail in section
Section 8.2.5.
Despite these drawbacks, RPI is widely used. For example, browser and
mobile clients typically use RPI to invoke services. It can also work well for
service-to-service communication, most notably in the API Composition
pattern described in chapter 10. Let’s look at the most common way to
implement RPI in a microservice architecture: REST.
8.2.2 Using REST
Today it’s fashionable to develop APIs in the RESTful style
(https://en.wikipedia.org/wiki/Representational_state_transfer). REST is an
IPC mechanism that (almost always) uses HTTP. Roy Fielding, the creator
of REST, defines it as follows:
REST provides a set of architectural constraints that, when applied as a
whole, emphasizes scalability of component interactions, generality of
interfaces, independent deployment of components, and intermediary
components to reduce interaction latency, enforce security, and
encapsulate legacy systems.
— roy.gbiv.com/pubs/dissertation/top.htm
A key concept in REST is a resource, which typically represents a single
business object, such as a Customer or Product, or a collection of business
objects. REST uses the HTTP verbs for manipulating resources, which are
referenced using a URL. For example, a GET request returns the
representation of a resource, which is often in the form of an XML document
or JSON object, although other formats such as binary can be used. A POST
request creates a new resource, and a PUT request updates a resource. The

Many developers claim their HTTP-based APIs are RESTful. But as Roy

understand why, let’s take a look at the REST maturity model.
THE REST MATURITY MODEL
Leonard Richardson (no relation to your author) defines a very useful
maturity model for REST
(http://martinfowler.com/articles/richardsonMaturityModel.html) that
consists of the following levels.
Level 0 — Clients of a level 0 service invoke the service by making
HTTP POST requests to its sole URL endpoint. Each request specifies
the action to perform, the target of the action (for example, the business
object), and any parameters.
Level 1 — A level 1 service supports the idea of resources. To perform
an action on a resource, a client makes a POST request that specifies
the action to perform and any parameters.
Level 2 — A level 2 service uses HTTP verbs to perform actions: GET
to retrieve, POST to create, and PUT to update. The request’s query
parameters and body, if any, specify the action’s parameters. This
enables services to use web infrastructure such as caching for GET
requests.
Level 3 — The design of a level 3 service is based on the terribly named
HATEOAS (Hypertext As The Engine Of Application State) principle.
The basic idea is that the representation of a resource returned by a
GET request contains links for performing actions on that resource. For
example, a client can cancel an order using a link in the representation
returned by the GET request that retrieved the order. The benefits of
HATEOAS include no longer having to hard-wire URLs into client
code (https://www.baeldung.com/java-hal-vs-hateoas).

BENEFITS AND DRAWBACKS OF REST
REST offers several important benefits. It is simple and familiar, making it

8.2.4 Implementing resilient synchronous communication
As described in Chapter 4, when one service, such as the API Gateway,
synchronously invokes another service, such as the Order Service, the two
services become runtime coupled. The API Gateway cannot complete its
request and respond to the client until the Order Service responds. This
dependency introduces the risk of increased latency and reduced availability.
If the Order Service has high latency or is unavailable, the API Gateway
may fail to meet its own SLOs. To address these risks, a service that makes
synchronous requests must apply resiliency patterns that limit the blast
radius of downstream failures and help maintain predictable performance.
Services must use the following resiliency patterns
(https://github.com/resilience4j/resilience4j?tab=readme-ov-file#resilience-

---

## Page 333

patterns) in order to tolerate high latency and failures while meeting their
SLOs:
Bulkhead - limit the number of outstanding calls to the service and fail
fast if the limit has been reached, which typically occurs when the
remote service is overloaded.
Cache - Reduce load and latency by using cached responses.
Circuit breaker - after repeated failures, fail fast for a timeout period
before attempting to invoke the remote service again
Fallback - Provide a degraded but useful response when the primary
path fails, such as default data or a previously cached value.
Rate limiter - limit the frequency of calls to avoid overloading the
remote service.
Retry - after a failure, automatically retry the call.
Time limiter - wait a timeout period for a response from the remote
service. If the maximum rate is exceeded, the attempt to call the service
fails. It ensures that the client does not violate its latency SLO or
consume valuable resources for too long.
Let’s discuss the Circuit breaker and Fallback patterns in more detail since
they are essential for preventing cascading failures and ensuring services can
still return useful responses during disruptions.
CIRCUIT BREAKER PATTERN
The Circuit breaker pattern behaves analogously to a circuit breaker in an
electrical circuit that interrupts the power flow when it detects excessive
current. Figure 8.7 shows the behavior of the circuit breaker. It tracks the
number of successful and failed requests, and if the error rate exceeds some
threshold, it trips the circuit breaker so that further attempts fail immediately
for a timeout period. Initially, the circuit breaker is closed which allows
requests to be made. But if there are too many failed requests - indicating
that the other service is temporarily unavailable and further calls are
pointless or even harmful - the circuit breaker trips and transitions to the
open state. For the duration of the timeout period, requests fail immediately.
After the timeout period, the circuit breaker transitions to the half-open state,

---

## Page 334

which allows through test requests. If those succeed, the circuit break
transitions to the closed state. Otherwise, it opens again.
Figure 8.7 A circuit breaker implements a state machine. Initially it is in the closed state, which

request to be made. If the test request succeeds, the circuit breaker transitions to the closed
state, otherwise it goes back to the open state.
The Circuit breaker pattern is useful because a large number of requests
failing suggests that the service is unavailable. It’s pointless for a client to
send more requests, which will ultimately fail. What’s more, those requests
might even contribute to the problem. It makes sense, therefore, to
temporarily stop sending requests to give the service a chance to recover.

OVERVIEW OF SERVICE DISCOVERY
Service discovery is the mechanism in a distributed system, such as a
microservice architecture, that enables services to find and communicate
with each other without needing hardcoded network locations. It’s
conceptually quite simple: its key component is a service registry, which is a
database of the network locations of an application’s service instances. The
service discovery mechanism updates the service registry when service

---

instances start and stop. When a client invokes a service, the service
discovery mechanism queries the service registry to obtain a list of available
service instances and routes the request to one of them.
Service discovery was once a major challenge in microservice architectures,
but modern platforms have made it largely automatic. Early systems used
application-level service discovery, which relied on applications querying
dedicated service registries such as Netflix Eureka, Consul, or Zookeeper
that tracked the IP addresses of running service instances. Today, service
discovery is provided by the deployment platforms, such as Kubernetes. As a
result, most teams no longer need to implement service discovery at the
application level. Let’s briefly look at application-level service discovery
before exploring platform-provided service discovery.

8.3 Communicating using the asynchronous

When using messaging, services communicate by asynchronously

8.3.1 Overview of messaging
An extremely useful model of messaging is defined in the book Enterprise
Integration Patterns (Addison-Wesley Professional, 2003) by Gregor Hohpe
and Bobby Woolf. In this model, messages are exchanged over message
channels. A sender (an application or service) writes a message to a channel,
and a receiver (an application or service) reads messages from a channel.
Figure 8.12 shows how messaging works. Messages are exchanged over

in the sender invokes a sending port interface, which encapsulates the
underlying communication mechanism. The sending port is implemented by
a message sender adapter class, which sends a message to a receiver via a
message channel. A message handler adapter class in the receiver is invoked
to handle the message. It invokes a receiving port interface implemented by
the consumer’s business logic. Any number of senders can send messages to
a channel. Similarly, any number of receivers can receive messages from a
channel.

ABOUT MESSAGES
A message consists of a header and a message body

is a collection of name-value pairs - metadata that describes the data being
sent. In addition to name-value pairs provided by the message’s sender, the
message header contains name-value pairs, such as a unique message id
generated by either the sender or the messaging infrastructure, and an
optional return address, which specifies the message channel that a reply
should be written to. The message body is the data being sent, in either text
or binary format.
There are four kinds of messages:
Document — A generic message that contains only data. The receiver
decides how to interpret it.
Event — A message indicating that something notable has occurred in
the sender. An event is often a domain event, which represents a state
change of a domain object such as an Order, or a Customer.
Command — A message that’s the equivalent of an RPC request. It
specifies the operation to invoke and its parameters.
Reply - the message sent in response to a command
The approach to the microservice architecture described in this book uses
commands and events extensively.

BENEFITS AND DRAWBACKS OF ASYNCHRONOUS MESSAGING
Asynchronous messaging has many advantages:
No runtime coupling - Services exchange messages without waiting for
a response from the other service. Unlike with a synchronous
request/response protocol such as HTTP, where both the client and
service must be available for the duration of the exchange, the sender
doesn’t care if the receiver is temporarily unavailable.
Workload management and buffering - The message broker buffers
messages until they can be processed, allowing the system to absorb
spikes in demand or handle slow consumers gracefully. For example, an
online store can continue accepting orders from customers even when
the order-fulfillment service is slow or temporarily offline.
Location independence — A client makes a request by simply sending a
message to the appropriate channel. The client is completely unaware
of the service instances. It doesn’t need to use a discovery mechanism
to determine the location of a service instance.
Flexible communication — Messaging supports all of the interaction
styles described earlier.

interprocess 
communication — RPC-based 
mechanisms
attempt to make invoking a remote service look the same as calling a
local service. But due to the laws of physics and the possibility of
partial failure, they’re in fact quite different. Messaging makes these
differences very explicit so developers aren’t lulled into a false sense of
security.
Asynchronous messaging also has some downsides:
Potential performance bottleneck — There is a risk that the message
broker could be a performance bottleneck. Fortunately, many modern
message brokers are designed to be highly scalable.

Potential single point of failure — It’s essential that the message broker

another infrastructure component that must be provisioned, configured,

8.3.2 Implementing the interaction styles using messaging
One of the valuable features of messaging is that it’s flexible enough to
support all the interaction styles described in section Section 8.1.1. Some
interaction styles are implemented directly by messaging, while others must
be implemented on top of it.
Let’s look at how to implement each interaction style, starting with
request/response and asynchronous request/response.
IMPLEMENTING REQUEST/RESPONSE AND ASYNCHRONOUS
REQUEST/RESPONSE
When a client and service interact using either request/response or
asynchronous request/response, the client sends a request and the service
sends back a reply. The difference between the two interaction styles is that
with request/response the client expects the service to respond immediately,
whereas with asynchronous request/response there is no such expectation.
Messaging is inherently asynchronous and so only provides asynchronous
request/response. But a client could block until a reply is received.
The client and service implement the asynchronous request/response style
interaction by exchanging a pair of messages. As figure 8.13 shows, the
client sends a command message, which specifies the operation to perform
and parameters, to a point-to-point messaging channel owned by a service.
The service processes the requests and sends a reply message, which
contains the outcome, to a point-to-point channel owned by the client.

The client must tell the service where to send a reply message and must
match reply messages to requests. Fortunately, solving these two problems
isn’t that difficult. The client sends a command message that has a reply
channel header. The service writes the reply message, which contains a
correlation id
(https://www.enterpriseintegrationpatterns.com/patterns/messaging/Correlati

channel. The client uses the correlation id to match the reply message with
the request.

is inherently asynchronous. In theory, a messaging client could block until it
receives a reply, but in practice the client will process replies
asynchronously. What’s more, replies are typically processed by any one of
the client’s instances.

Messaging has built-in support for the publish/subscribe style of interaction.

multiple consumers. As described in chapters 9 and 5, services use
publish/subscribe to publish domain events, which represent changes to
domain objects. The service that publishes the domain events owns a

class. For example, the Order Service publishes Order events to an Order
channel, and the Delivery Service publishes Delivery events to a
Delivery channel. A service that’s interested in a particular domain object’s
events only has to subscribe to the appropriate channel.
IMPLEMENTING PUBLISH/ASYNC RESPONSES
The publish/async responses interaction style is a higher level style of
interaction that’s implemented by combining elements of publish/subscribe
and request/response. A client publishes a message, which specifies a reply
channel header, to a publish-subscribe channel. A consumer sends a reply
message containing a correlation id to the reply channel. The client gathers
the responses by using the correlation id to match the reply messages with
the request.
Each service in your application that has an asynchronous API will use one
or more of these implementation techniques. A service that has an
asynchronous API for invoking operations will have a message channel for
command messages. Similarly, a service that publishes events will publish
them to an event message channel.

8.3.4 Evolving an asynchronous API
Asynchronous messaging APIs, such as those that publish domain events or
implement command/reply interactions, inevitably change as a system and
its domain model evolve. Consider, for example, an Order Service that
publishes events, such as Order Created, and Order Fulfilled. Examples
of seemingly simple changes include adding attributes to events, such as
adding a discountCode to the Order Created event. The event types
themselves might also change. For example, to improve how the business is
modelled, Order Fulfilled could be replaced by Order Ready For Pickup
and Order Picked Up. Just as REST or gRPC APIs must evolve without
breaking existing clients, asynchronous APIs must also evolve without
disrupting existing consumers of messages.
Before we look in more detail at the different ways asynchronous APIs can
evolve, it’s important to understand one key difference between synchronous
and asynchronous communication. Unlike REST or gRPC APIs, where
clients and services interact in real time, messages in an asynchronous
system can outlive the version of the service that produced them.
PERSISTENCE AND MESSAGE LONGEVITY
Evolving a message-based API is more complex than changing a
synchronous one. In a request/response interaction, a client only exchanges
messages with the current version of a service, so compatibility concerns are
limited to services that are running at the same time. In contrast, messages in
an asynchronous architecture are persistent. They can remain in message
channels long after they were sent, especially events. A consumer might
process a message hours, days, or even months after it was sent. As a result,
new versions of services must be able to process messages produced by
older versions.
Let’s now look at a few different ways that events can evolve.

EVENTS

This looks like a straightforward change since an old consumer could simply

persistent, an updated consumer must be capable of handling older events

by the field’s type. But when using AVRO, the new schema must define a

Another common change is evolving the event types. For example, as

the schema simultaneously.

changed to consume the Order Ready For Pickup and Order Picked Up

events.

ecosystem of libraries and tools for creating, validating, and consuming

sent to or received from. The operation object also defines the channel’s

can optionally specify one or more reply messages that are sent in response,

8.3.6 Pattern: Ordered competing receivers
One challenge with implementing a service that receives messages is how to
increase throughput by horizontally scaling message receivers while
preserving message ordering. It’s common to need multiple instances of a
service to process messages concurrently. Moreover, even a single service
instance will probably use threads to concurrently process multiple
messages. Using multiple threads and service instances to concurrently
process messages increases the throughput of the application. But the
challenge with processing messages concurrently is ensuring that each
message is processed once and in the correct order.
For example, imagine three instances of a service reading from the same
point-to-point channel, where a sender publishes Order Created, Order
Updated, and Order Cancelled event messages sequentially. A simplistic
messaging implementation could deliver each of those messages
concurrently to a different receiver. Because of delays due to network issues
or garbage collection, messages might be processed out of order, leading to
inconsistent or invalid application state. In theory, a service instance might
process the Order Cancelled message before another service processes the
Order Created message!
A common solution used by modern message brokers, such as Apache Kafka
and AWS Kinesis, to scale consumers while preserving ordering is to use
sharded (partitioned) channels. Figure 8.15 shows how this works. The
solution has three parts:
1. A sharded channel consists of two or more shards, each of which
behaves like an independent channel.

---

## Page 365

2. The sender specifies a shard key in the message’s header, which is
typically an arbitrary string or sequence of bytes. The message broker
uses the shard key to assign the message to a particular shard/partition.
It might, for example, select the shard by computing a hash of the shard
key modulo the number of shards.
3. The messaging broker groups together multiple instances of a receiver
and treats them as the same logical receiver. Apache Kafka, for
example, uses the term consumer group. The message broker assigns
each shard to a single receiver. Shards are reassigned dynamically as
receivers start up or shut down.
Figure 8.15 Scaling consumers while preserving message ordering by using a sharded
(partitioned) message channel. The sender includes the shard key in the message. The message
broker writes the message to a shard determined by the shard key. The message broker assigns
each partition to an instance of the replicated receiver.
In this example, each Order event message uses its orderId as the shard key.
Each event for a particular order is published to the same shard, which is
read by a single consumer instance. As a result, those messages are
guaranteed to be processed in order.
8.3.7 Pattern: Idempotent consumer pattern
Another challenge to address when using messaging is dealing with
duplicate messages. A message broker should ideally deliver each message
only once, but guaranteeing exactly - once delivery is usually too costly.
What’s more, doing so would require the message broker and service’s
database to participate in a distributed transaction. As a result, most message
brokers guarantee to deliver a message at least once.

---

## Page 366

When the system is working normally, a message broker that guarantees at-
least-once delivery will deliver each message only once. But a failure of a
client, network, or message broker can result in a message being delivered
multiple times. Say a client crashes after processing a message and updating
its database — but before acknowledging the message. The message broker
will deliver the unacknowledged message again, either to that client when it
restarts or to another instance of the same service.
Ideally, you should use a message broker that preserves ordering when
redelivering messages. Imagine that the client processes an Order Created
event followed by an Order Cancelled event for the same Order, and that
somehow the Order Created event wasn’t acknowledged. The message
broker should redeliver both the Order Created and Order Cancelled
events. If it only redelivers the Order Created event, the client may undo
the canceling of the Order.
You can handle duplicate messages in one of two ways:
Write idempotent message handlers
Track messages and discard duplicates
Let’s look at each option.
WRITING IDEMPOTENT MESSAGE HANDLERS
If the application logic that processes messages is idempotent, then duplicate
messages are harmless. Application logic is idempotent if calling it multiple
times with the same input values has no additional effect. For example,
canceling an already cancelled order is an idempotent operation. So is
creating an order with a client-supplied id. An idempotent message handler
can be safely executed multiple times, provided that the message broker
preserves ordering when redelivering messages.
Unfortunately, application logic is often not idempotent, or you may be
using a message broker that doesn’t preserve ordering when redelivering
messages. In this situation, to prevent bugs, you must write message
handlers that track messages and discard duplicate messages.

---

## Page 367

TRACKING MESSAGES AND DISCARDING DUPLICATES
Consider, for example, a message handler that authorizes a consumer’s credit
card. It must authorize the card exactly once for each order. This is an
example of application logic that has a different effect each time it’s
invoked. If duplicate messages caused this message handler to execute
multiple times, the application would behave incorrectly. The message
handler that executes this kind of application logic must achieve
idempotence by detecting and discarding duplicate messages.
A simple solution is for a message consumer to track the messages that it has
processed using the message id and discard any duplicates. It could, for
example, store the message id of each message it consumes in a database
table. Figure 8.16 shows how to do this using a dedicated table.
Figure 8.16 A consumer detects and discards duplicate messages by recording the IDs of

Another option is for a message handler to record message IDs in an

useful when using a NoSQL database that has a limited transaction model

8.3.8 Pattern: Transaction outbox
A service often needs to publish messages as part of a transaction that
updates the database. For instance, throughout this book you will see
examples of services that publish domain events whenever they create or
update business entities. Both the database update and the sending of the
message must happen within a transaction. Otherwise, a service might
update the database and then crash, for example, before sending the
message. If the service doesn’t perform these two operations atomically, a
failure could leave the system in an inconsistent state.
The traditional solution is to use a distributed transaction that spans the
database and the message broker. But as you’ll learn in chapter 9, distributed
transactions aren’t a good choice for modern applications. Moreover, many
modern brokers such as Apache Kafka don’t support distributed transactions.
As a result, an application must use a different mechanism to atomically
update the database and publish messages. Let’s look at how that works.
USING A DATABASE TABLE AS A MESSAGE QUEUE
Let’s imagine that your application is using a relational database. A
straightforward way to reliably publish messages is to apply the
Transactional outbox pattern. This pattern uses a database table as a
temporary message queue. As figure 8.17 shows, a service that sends
messages has an OUTBOX database table. As part of the database transaction
that creates, updates, or deletes business objects, the service sends messages
by inserting them into the OUTBOX table. Atomicity is guaranteed because this
is a local ACID transaction.

8.4 Summary
The microservice architecture is a distributed architecture, so
interprocess communication plays a key role.

---

## Page 382

There are numerous IPC technologies, each with different trade-offs.
One key design decision is to choose either a synchronous remote
procedure invocation pattern or the asynchronous messaging pattern.
Synchronous remote procedure invocation-based protocols, such as
REST, are the easiest to use. But services should ideally communicate
using asynchronous messaging in order to reduce runtime coupling.
In order to prevent failures from cascading through a system, a service
client that uses a synchronous protocol must be designed to handle
partial failures, which is when the invoked service is either down or
exhibiting high latency. In particular, it must use resiliency patterns
including the Circuit breaker pattern.
An architecture that uses synchronous protocols must include a service
discovery mechanism in order for clients to determine the network
location of a service instance. The simplest approach is to use the
service discovery mechanism implemented by the deployment platform.
But an alternative approach is to implement service discovery at the
application level. It’s more work, but it does handle the scenario where
services are running on multiple deployment platforms.
A good way to design a messaging-based architecture is to use the
messages and channels model, which abstracts the details of the
underlying messaging system. You can then map that design to a
specific messaging infrastructure, which is typically message broker-
based. Each channel’s requirements - ordering requirements, delivery
guarantees, and so on - determine the choice of messaging technology.
There are three important messaging patterns:
Transaction outbox - ensures that the sender atomically updates the
database and sends a message. It first writes the message to the
database as part of the database transaction. A separate process
then retrieves the message from the database using either the
Polling publisher pattern or the Transaction log tailing pattern and
sends it to the message broker.
Idempotent Consumer pattern - detects and discards duplicate
messages by tracking processed messages in the database. It’s
important to use this pattern when a message handler is not
naturally idempotent since a message broker can deliver a message
multiple times.

---

## Page 383

Competing consumers - increases throughput by processing
concurrently while still preserving ordering.
A good way to reduce runtime coupling is for a service to respond
immediately to a synchronous request, such as an HTTP POST, and
complete the remaining work asynchronously by exchanging messages
with other services.
It’s essential to carefully manage the evolution of a service’s API.
Backward-compatible changes are the easiest to make because they
don’t impact clients. If you make a breaking change to a service’s API,
it will typically need to support both the old and new versions until its
clients have been upgraded.
It’s important to define a service’s API using an Interface Definition
Language (IDL), such as OpenAPI for REST APIs, or AsyncAPI for
messaging-based APIs. An IDL defines the API in a precise, machine-
readable format that supports code generation, documentation, and
validation.
The runtime behavior of a service API - latency, availability, and related
properties - is described by service level objectives (SLOs) and, in
some cases, a service level agreement (SLA). An SLO is a target for a
service level indicator, such as availability. An SLA is a contractual
commitment to one or more SLOs, often with penalties or credits for
violations.
An error budget is the allowable amount of unreliability a service can
experience without breaching its SLOs. Error budgets guide whether a
team should focus on feature development or ensuring that their service
meets its SLOs.
An important part of an API client is the DTOs representing the
messages that are exchanged with the service. These DTOs can be
written by hand - often more time-consuming but can reduce design-
time coupling - or code generated from the IDL, which is convenient
but can unnecessarily design-time couple the client to the entire API.
Clients for asynchronous APIs should use a library or framework that
abstracts the low-level details of the messaging infrastructure and
directly supports higher-level interaction styles, such as command/reply
and domain event publishing. The open-source Eventuate platform is an
example of a framework that provides these capabilities. It also
