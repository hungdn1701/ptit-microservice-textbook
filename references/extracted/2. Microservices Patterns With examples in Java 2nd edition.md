A service is just one part of an application, and so it has a much smaller code

developer’s laptop. A developer can get feedback quickly and easily without
needing to wait for the deployment pipeline to complete, or worse, having to
ship their code to a dedicated testing environment before running any tests.

SEPARATE DEPLOYMENT OF APPLICATION MODULES WITH
DIFFERENT CHARACTERISTICS
Another benefit of the microservice architecture is that application modules
with different characteristics can be packaged as separate services in order to
improve architectural properties, such as scalability, availability, security and
ease of development. You could, for instance, improve the availability of
critical functionality by putting its implementation in its own service that
runs on its own high availability infrastructure, which isolates it from
changes to the rest of the application. I describe how other architectural
properties can be improved by segregating modules by characteristics in
chapter 7.
1.3.4 No such thing as a silver bullet: the drawbacks of
microservices
As Fred Brooks said back in 1986 (No Silver Bullet Essence and Accidents
of Software Engineering by Fred Brooks) "There is no silver bullet". Like
every other technology, the microservice architecture has both benefits and
drawbacks. The microservice architecture has five key drawbacks:
Distributed operations can be more complicated to design
There is the risk of creating a distributed monolith
Designing a good microservice architecture requires a level of rigor and
discipline that many organizations are unaccustomed too
The microservice architecture requires the rest of the fast flow success
triangle
Embracing independent deployability is a significant change
Let’s look at each of these drawbacks in more detail, starting with how
distributed operations can be more complicated to design.
DISTRIBUTED OPERATIONS CAN BE COMPLICATED TO DESIGN
Some operations in a microservice architecture will span multiple services.

Figure 1.13 Some, but not all, operations in a microservice architecture will span multiple

it’s unreliable, and not all services are guaranteed to be simultaneously available. You must also

The first problem is that since services are process boundaries, they must
communicate over the network. You often need to handle failure modes that
don’t occur within a monolith. For example, the Order Service must handle
the scenario where the Customer Service is unavailable.
Also, as stated by the Fallacies of Distributed Computing
(https://en.wikipedia.org/wiki/Fallacies_of_distributed_computing), the
network isn’t free. You must ensure that distributed operations are efficient.
You must also avoid introducing runtime coupling, which occurs when
services call each other synchronously. Runtime coupling is a problem
because it reduces availability and increases latency. Careful design is

required to avoid creating a Christmas tree light architecture

cascades through the system and causes a production outage.
The second problem is that for reasons I’ll discuss in chapter 7, operations,
such as createOrder(), that span multiple services cannot be implemented
as a single database transaction. Instead, you must implement it using an
unfamiliar and more complex transaction model known as eventual
consistency. An eventually consistent operation consists of a series of
transactions in each of the services. In this model, you must sometimes write
undo logic and handle concurrency explicitly, rather than simply relying on
the database to manage it for you.
THERE IS THE RISK OF CREATING A DISTRIBUTED MONOLITH
There’s also the risk of creating a distributed monolith, which consists of
tightly design-time coupled services that change in lockstep. It is an anti-
pattern that combines the worst aspects of the monolithic and microservice
architectures. The problem is that you cannot simply make a breaking
change to a service’s API because that will cause a production outage.
Instead, you must carefully and incrementally evolve an API in ways that
don’t break the service’s clients. It can be a lot of work and require a lot of
coordination between the teams that own the affected services.
DESIGNING A MICROSERVICE ARCHITECTURE REQUIRES
RIGOR AND DISCIPLINE
Fortunately, we can avoid many of the problems of a distributed architecture
by carefully defining the service boundaries. We need to be vigilant to
ensure that the services are loosely design-time coupled. We can also, for
example, design the services so that every performance critical operation is
implemented by a single service. This might mean that the services are larger
than you would normally want, but quite often it’s a worthwhile trade-off.
The challenge, however, is designing a good microservice architecture
requires rigor and discipline that many organizations are unaccustomed to.

There’s a tendency to not think through design decisions, especially when
deadlines are looming. After all, in a monolithic application, you can simply
add a new class anywhere in the code base and the application will work.
Unfortunately, such a haphazard approach to software design doesn’t work
in a microservice architecture.
For example, one common problem is the More the Merrier anti-pattern
where an organization takes the 'micro' in microservices far too literally and
blindly adds services. Unfortunately, this style of development results in an
overly complex, fine-grained architecture that’s difficult to understand and
maintain. The architecture is also inefficient because of the overhead of the
network calls. And it’s brittle because of the runtime coupling between the
services.
Design decisions about which service should implement a feature and
whether to add a new service are extremely consequential. In particular, as I
mentioned earlier in Section 1.3.1, a service should exist only if it solves an
actual problem. You need to consider numerous trade-offs when designing a
microservice architecture. In chapter 20, I describe Assemblage, which is a
design process for doing exactly that. And, throughout this book, I describe
various patterns from the Microservice Pattern Language that can help you
design a good microservice architecture.
THE CONSEQUENCES OF LOTS OF LITTLE SERVICES
I once met with a team that was having problems testing their services. I
asked my usual questions: "How many services do you have?", "How
many developers do you have?". The answer to both questions was five.
My advice was to simplify their architecture by merging the services into
a single service. As an added benefit, it solved their testing problem.
THE MICROSERVICE ARCHITECTURE REQUIRES THE REST OF
THE FAST FLOW SUCCESS TRIANGLE

EMBRACING INDEPENDENT DEPLOYABILITY IS A SIGNIFICANT
CHANGE
The fourth challenge is that organizations often struggle to embrace the
concept of independently deployable services. A key benefit of microservice
architecture is that each service has its own deployment pipeline. Yet some
organizations struggle to adopt this concept. They insist on running end-to-
end tests of the entire application before deploying an updated service.
The problem is that end-to-end tests involve a large number of moving parts
which makes them slow and prone to random failures. Manual review is
required to determine whether a change is releasable. End-to-end tests
prevent the organization from achieving the goal of deploying changes
quickly and easily. The solution is to rethink the approach to testing and
move end-to-end tests out of the deployment pipeline.
1.4 When to use the microservice architecture
If you are contemplating adopting the microservice architecture, the first
thing to remember is that the monolithic architecture is a perfectly good
choice for many applications. As described earlier, it’s simple, easy to
understand, and easy to develop - at least when the application is small.
Consequently, you should not automatically choose the microservice
architecture for your application. Instead, you should choose the architecture
that best fits your application’s requirements. Or to put it another way, you
should pick the architecture whose benefits outweigh its drawbacks for your
application.

There are two scenarios when you need to decide between the monolithic
architecture and the microservice architecture. The first scenario is when
building a brand-new application. The second is when you have a monolithic
application that you feel has own outgrown the monolithic architecture. In
both scenarios, careful analysis is required to determine whether the
microservice architecture is the right choice. Let’s look at each of these
scenarios in more detail, starting with the greenfield scenario when you build
an application from scratch.
1.4.1 Building an application from scratch: typically a monolith-
first approach
Let’s imagine that you are starting the development of a brand-new
application using a list of the latest, shiny new technologies. You might be
tempted to include the microservice architecture in that list of modern
technologies. The reality is, however, that it’s unlikely that you have the fast
flow-related problems that the microservice architecture addresses. You
haven’t written a line of code, so your application won’t be large and
complex for some time, maybe never. You don’t have a large organization,
so you don’t need to worry about team autonomy. You can, therefore,
achieve fast flow with a monolithic architecture.
While fast flow is not normally a motivation for starting with the
microservice architecture, it’s possible that some of the benefits outlined in
Section 1.3.3 might be. If your application needs to be written using a
mixture of technologies, then a simple microservice architecture consisting
of a few services makes sense. Or perhaps, you can address significant
scalability, availability and security problems by segregating modules that
have different characteristics in their own services.
Even if the microservice architecture is a good fit for your application, you
risk creating a distributed monolith. You need stable APIs in order for
services to be loosely design-time coupled. But when you are building a
brand-new application, it’s unlikely that you understand the problem domain
well enough to design stable APIs. You will often need to iterate on
boundaries and APIs before they become stable. It’s far easier to do that in a
monolithic architecture than in a microservice architecture. Of course, there

---

## Page 39

are exceptions, such when you must use multiple technologies or there are
compelling reasons to segregate modules, but I strongly recommend a
monolith-first approach.
1.4.2 Refactoring a monolith to the microservice architecture
The second scenario for adopting the microservice architecture is when your
application has outgrown its monolithic architecture. Perhaps, the pace of
software development has slowed down Deployments have become less
frequent. Or, teams are spending increasing amounts of time in meetings
coordinating changes.
On the surface, these are all signs that to achieve fast flow you need to adopt
the microservice architecture. But before you make that decision, you must
be sure that the monolithic architecture is the root cause of the slow software
development. And, if it is, you must migrate to microservices in a way that
minimizes risk and maximizes the delivery of value. Let’s first look at how
to understand the root cause of slow software development.
UNDERSTAND THE ROOT CAUSE OF SLOW SOFTWARE
DEVELOPMENT
The challenge that you face in this scenario, however, is being sure that the
monolithic architecture is preventing fast flow. Perhaps you simply need to
modularize the monolith so that teams own loosely design-time coupled
modules. Or perhaps, the problem is that you have not properly implemented
the other elements of the fast flow success triangle: DevOps and Team
Topologies. You need to adopt DevOps as your way of working and Team
Topologies as your organizational structure. You also need an automated
deployment pipeline. If and only if these changes are insufficient to achieve
fast flow should you consider migrating to the microservice architecture.
INCREMENTALLY MIGRATE TO THE MICROSERVICE
ARCHITECTURE
If you decide to migrate your application to the microservice architecture,
the next decision you must make is how and where to start. What not to do is

---

## Page 40

big bang rewrite, where you write a brand new microservices-based
application from scratch. While it’s always tempting to start from scratch
since you can ignore the existing mess, it’s also risky. You don’t deliver
anything of value until the new application is complete, which is likely to far
in the future. Also, you don’t get any feedback on any of the design
decisions you made until the distant future. As Martin Fowler is rumored to
have said, "A big bang rewrite often results in a big bang".
As I describe in chapter 21, a much better approach is to use the Strangler
Fig pattern, which incrementally migrates functionality from the monolith to
the microservice architecture. Not only are you delivering value early and
often, but you are also getting constant feedback on your design decisions.
You can also implement new features that the business needed yesterday
directly as services. It’s not always easy to do because you are breaking apart
a legacy code base, but it’s almost always better than the big bang rewrite.
1.5 Summary
In order to thrive in today’s world, businesses need fast flow, which is
the continuous delivery of small, valuable changes to customers and
continuous stream of feedback on each change to back to development.
Fast flow requires the success triangle: a combination of DevOps, Team
Topologies and an architecture that enables them by promoting team
autonomy and fast deployment pipelines.
DevOps is a set of principles and practices for delivering software
quickly and reliably.
Team Topologies is a set of principles and practices for designing
organizations to enable fast flow.
A monolithic architecture consists of a single codebase that’s deployed
as a single deployable unit, such as an executable JAR or WAR file, by
a single deployment pipeline.
While the monolithic architecture is not an anti-pattern, some
applications, especially those that are large and are developed by large
teams, must typically use the microservice architecture in order to
achieve fast flow.

Many organizations have enthusiastically adopted the microservice

2.1 FTGO architecture V1: The eventual descent
into monolithic hell
Like many other applications that were developed in the early 2000s, the
first version of the FTGO application was a monolith. It consisted of a single
Java Web Application Archive (WAR) file. Initially, the monolithic
architecture worked well. It enabled FTGO to grow by leaps and bounds and
become one of the leading online food delivery companies in the United
States.
But as the business grew, so did the application and the engineering
organization. The engineering organization grew from a single team to over
twenty teams. The application became large and complex. Despite the best
efforts of the FTGO engineering organization, it became an example of the
Big Ball of Mud pattern (http://www.laputan.org/mud/). To quote Foote and
Yoder, the authors of that pattern, it’s a "haphazardly structured, sprawling,
sloppy, duct-tape and bailing wire, spaghetti code jungle." As a result,
development slowed down to the point that it even jeopardized FTGO’s
plans to expand overseas.
This section starts by describing the original version of the FTGO
architecture. Then it talks about why the monolithic architecture worked
well initially. After that, we’ll get into how the FTGO application outgrew its
architecture and how development became hell.
2.1.1 The architecture of the FTGO application
At its core, the FTGO application is quite simple. Consumers use the FTGO
website or mobile application to place food orders at local restaurants.
FTGO coordinates a network of couriers who deliver the orders. It’s also
responsible for paying couriers and restaurants. Restaurants use the FTGO
website to edit their menus and manage orders. The application uses various
web services, including Stripe for payments, Twilio for sending messages,

2.1.2 Why the monolithic architecture worked well initially
The FTGO developers enjoyed several benefits of the monolithic

Simple to develop — IDEs and other developer tools are focused on
building a single application.
Easy to make radical changes to the application — You can change the
code and the database schema, build, and deploy.
Straightforward to test — The developers wrote end-to-end tests that
launched the application, invoked the REST API, and tested the UI with
Selenium.
Straightforward to deploy — All a developer had to do was copy the
WAR file to a server that had Tomcat installed.
Easy to scale — FTGO ran multiple instances of the application behind
a load balancer.

Another problem with the FTGO application was the lack of reliability.
Testing the application thoroughly was difficult due to its large size. This
lack of testability means bugs made their way into production. As a result,
there were frequent production outages.
What’s more, the application lacked fault isolation because all modules were
running within the same process. Every so often, a bug in one module — for
example, a memory leak — crashed all instances of the application, one by
one. To make matters worse, since production was updated infrequently,
each deployment consisted of a large number of changes. Sometimes
identifying the root cause was like looking for a needle in a haystack and
took a long time.
The FTGO developers didn’t enjoy being paged in the middle of the night
because of a production outage. The business people liked the loss of

SCALING WAS CHALLENGING
The FTGO team also had problems scaling its application. That’s because

of CPU. Because these modules are part of the same application, FTGO
needed to compromise on the server configuration.
EVOLVING THE TECHNOLOGY STACK WAS DIFFICULT
The final serious problem experienced by the FTGO team was that keeping
the application’s technology stack current was extremely challenging. That’s
because upgrading the JVM and the application’s dependencies—
frameworks and libraries—often impacted the entire application. Some
upgrades required the focus of the entire engineering organization. The
business was reluctant to approve such upgrades because they delayed the
delivery of new features, sometimes for unexpectedly long periods of time.
As a result, the application’s technology stack was often several versions
behind. And, to make matters worse, there were periodically major panics
when security vulnerabilities required urgent upgrades.
Another technology stack related problem was that the application’s
monolithic architecture meant that the organization was locked into the
JVM. FTGO developers could not conduct realistic experiments with non-
JVM languages such as GoLang or Rust to determine whether they were
worth adopting. Fortunately, the JVM ecosystem has continued to thrive, so
this is not as big a problem as it could have otherwise been.

2.2 FTGO architecture V2: Fine-grained
microservices everywhere - a different kind of hell
After attending a software architecture conference in early 2016, Mary - the
CTO of Food To Go - came to two important conclusions. First, she
concluded that FTGO was suffering from a classic case of monolithic hell.
The application and engineering organization had outgrown the monolithic
architecture. Second, she decided that the microservice architecture was the
solution. FTGO need to migrate their application from the monolithic
architecture to the microservice architecture.
Mary and her team scoured the web for information about the microservice
architecture. They watched numerous talks, read many blog posts, and
bought several books. Mary also talked to members of her network about
their experiences adopting microservices. Armed with what they thought was
a good understanding of the microservice architecture, Mary and the FTGO
engineering organization enthusiastically adopted the microservice
architecture.
This section begins by describing version 2 of the FTGO architecture, which
is based on microservices. It then explains where the FTGO engineers went
wrong—including taking the definition of MICROservices too literally—and
the numerous problems that resulted.
2.2.1 How FTGO adopted the microservice architecture
The FTGO engineers adopted microservices in two different ways. First,
they started projects to rewrite their existing applications from scratch using
the microservice architecture. Those applications consist of the main FTGO

continuous stream of small changes into production. But sadly, the
deployments were still infrequent. And, when they did occur, the changes

application rather than just the service that had changed. They used running
slow and brittle end-to-end tests of the entire application rather than just the
needed to test the entire application.

amount of manual testing. This manual testing was time-consuming and
prone to human error, leading to delays and inconsistencies. As a result,
deployments occurred once every week or two.

often, debugging a production outage was like looking for a needle in a
haystack. It took time to identify the root cause and fix it.

3.2 Software architecture is multidimensional
In the same way that a building’s architecture can be viewed from multiple
perspectives - structural, plumbing, electrical, etc. - so can an application’s
architecture. An important concept in the field of software architecture is an
N+1 view model. A view model us an excellent way to describe an

---

## Page 73

application’s architecture. It consists of N views, each of which describes an
important static aspect - or dimension - of the architecture. A static view
consists of elements and relations. In addition to the multiple elements and
relations views, the view model also has a '+1' scenario view. The scenario
view consists of scenarios that describe the application’s dynamic behavior
by showing how the static views' elements interact.
THE ORIGINAL 4+1 VIEW MODEL
Phillip Krutchen proposed the concept of a view model in his classic
paper "Architectural Blueprints—The '4+1' View Model of Software
Architecture" (https://www.cs.ubc.ca/~gregor/teaching/papers/4+1view-
architecture.pdf). The paper describes the 4+1 view model, which consists
of the following four views:
logical - the object model consisting of classes and their relationships
that are created by developers
implementation - the application’s code compiled/packaged into
modules (e.g., JAR files) and components (e.g., executable JARs or
WAR files)
process - the application’s components as a set of running processes

3.2.1 Domain view

is to implement the application’s business domains. The domain view
consists of the programming language elements that developers write and the
relationships between them. It organizes those programming language
elements into subdomains. A subdomain is a slice of business functionality
and is owned by a small team. In an object-oriented language, such as Java,
a subdomain consists of classes, which are organized into a hierarchy of
packages.
The domain view’s elements have many types of relationships between
them. These relationships include association and inheritance relationships
between classes. There are also Whole-Part relationships between classes,
packages, and subdomains: a package contains classes and sub-packages;
and a subdomain contains packages. There is also the uses relationship
between subdomains since classes in one subdomain might invoke classes in
another.
A system operation is implemented by one or more subdomains. The
operation is first handled by one subdomain, which then might collaborate
with other subdomains. For example, the createOrder() operation is first
handled by the Order Management subdomain. It then uses the Consumer
Management subdomain to verify that the consumer exists and can place
orders and the Delivery Management subdomain to schedule a delivery.

3.2.4 Build view
The Build view describes the path that code takes from the developer’s

Deployment pipelines - build and test components using a build tool
and deploy them into production.

it might also be initiated by the successful completion of a different pipeline.
It invokes the build tool one or more times to build and test the component
or components. Once the tests for the components pass, the deployment
pipeline deploys the updated components to production. As I described in
chapter 1, a deployment pipeline needs to be fast in order to provide
developers with fast feedback that’s required by fast flow.
One way to accelerate the build is to use incremental builds. A build tool that
supports incremental builds reuses the results of previous executions of the
deployment pipeline in order to build and test only those software elements
affected by a change. As a result, the deployment pipeline is much faster
than if it were to build and test everything scratch.

which tasks are up to date and skips them.

architecture because of their technical requirements. For the Food to Go

significant scenarios could be those that either process large amounts of data

3.3 Overview of architectural styles
In the physical world, a building’s architecture often follows a particular
style, such as Victorian, American Craftsman, or Art Deco. Each style is a
package of design decisions that constrains a building’s features and building

---

## Page 94

materials. The concept of architectural style also applies to software. David
Garlan and Mary Shaw, pioneers in the discipline of software architecture,
define an architectural style as follows:
An architectural style, then, defines a family of such systems in terms of
a pattern of structural organization. More specifically, an architectural
style determines the vocabulary of components and connectors that can
be used in instances of that style, together with a set of constraints on
how they can be combined.
— David Garlan and Mary Shaw
An Introduction to Software Architecture
A particular architectural style provides a limited palette of elements
(components) and relations (connectors) from which you can define a view
of your application’s architecture. An application typically uses a
combination of architectural styles.
3.3.1 Layered architecture
The classic example of an architectural style is the layered architecture. A
layered architecture organizes software elements into layers. Each layer has
a well-defined set of responsibilities. A layered architecture also constrains
the dependencies between the layers. A layer can only depend on either the
layer immediately below it (if strict layering) or any of the layers below it.
You can apply the layered architecture to any of the four views discussed
earlier. The popular three-tier architecture is the layered architecture applied
to the domain view. It organizes the application’s classes into the following
tiers or layers:
Presentation layer — Contains code that implements the user interface
or external APIs
Business logic layer — Contains the business logic
Persistence layer — Implements the logic of interacting with the
database

---

## Page 95

The layered architecture is a great example of an architectural style, but it
does have some significant drawbacks:
Single presentation layer — It doesn’t represent the fact that an
application is likely to be invoked by more than just a single system.
Single persistence layer — It doesn’t represent the fact that an
application is likely to interact with more than just a single database.
Defines the business logic layer as depending on the persistence layer 
— In theory, this dependency prevents you from testing the business
logic without the database.
Also, the layered architecture misrepresents the dependencies in a well-
designed application. The business logic typically defines an interface or a
repository of interfaces that define data access methods. The persistence tier
defines DAO classes that implement the repository interfaces. In other
words, the dependencies are the reverse of what’s depicted by a layered
architecture.
Let’s look at an alternative architecture that overcomes these drawbacks: the
hexagonal architecture.
3.3.2 About the hexagonal architecture style
Hexagonal architecture is an alternative to the layered architectural style. As
figure 3.16 shows, the hexagonal architecture style organizes the logical
view in a way that places the business logic at the center. Instead of the
presentation layer, the application has one or more inbound adapters that
handle requests from the outside by invoking the business logic. Similarly,
instead of a data persistence tier, the application has one or more outbound
adapters that are invoked by the business logic and invoke external
applications. A key characteristic and benefit of this architecture is that the
business logic doesn’t depend on the adapters. Instead, the adapters depend
upon it.

An outbound adapter implements an outbound port and invokes an external system.
The business logic has one or more ports. A port defines a set of operations
and is how the business logic interacts with what’s outside of it. In Java, for
example, a port is often a Java interface. There are two kinds of ports:
inbound and outbound ports. An inbound port is an API exposed by the
business logic, which enables it to be invoked by external applications. An
example of an inbound port is a service interface, which defines a service’s
public methods. An outbound port is how the business logic invokes external
systems. An example of an output port is a repository interface, which
defines a collection of data access operations.
Surrounding the business logic are adapters. As with ports, there are two
types of adapters: inbound and outbound. An inbound adapter handles
requests from the outside world by invoking an inbound port. An example of
an inbound adapter is a Spring MVC Controller that implements either a set
of REST endpoints or a set of web pages. Another example is a message

---

## Page 97

broker client that subscribes to messages. Multiple inbound adapters can
invoke the same inbound port.
An outbound adapter implements an outbound port and handles requests
from the business logic by invoking an external application or service. An
example of an outbound adapter is a data access object (DAO) class that
implements operations for accessing a database. Another example would be
a proxy class that invokes a remote service. Outbound adapters can also
publish events.
An important benefit of the hexagonal architectural style is that it decouples
the business logic from the presentation and data access logic in the
adapters. The business logic doesn’t depend on either the presentation logic
or the data access logic. Because of this decoupling, it’s much easier to test
the business logic in isolation. For example, tests can mock outbound
adapters and invoke the business logic in the same way an inbound adapter
would.
Another benefit is that it more accurately reflects the architecture of a
modern application. The business logic can be invoked via multiple adapters,
each of which implements a particular API or UI. The business logic can
also invoke multiple adapters, each one of which invokes a different external
system. Hexagonal architecture is a great way to describe the architecture of
each service in a microservice architecture.
The layered and hexagonal architectures are both examples of architectural
styles. Each defines the building blocks of an architecture and imposes
constraints on the relationships between them. The hexagonal architecture
and the layered architecture, in the form of a three-tier architecture, organize
the logical view.

3.4 Architecting: what is it, who does it and when?
An application’s architecture is an artifact created by the process of
architecting. In this section, I’ll describe what architecting is, who does it,
and when it’s done. Let’s start with what architecting is.
3.4.1 Architecture as a series of design decisions
An interesting way to think about architecture and architecting is described
by the paper Software Architecture as a Set of Architectural Design
Decisions* by Anton Jason and Jan Bosch
(https://www.researchgate.net/publication/220864796_Software_Architectur
e_as_a_Set_of_Architectural_Design_Decisions). The authors view, as the
title suggests, "architecture as a composition of a set of explicit design
decisions". By implication, architecting is the process of making those
design decisions.
As figure 3.17 shows, a design decision consists of the following elements:
Problem - the problem to solve
Motivation - why the problem is a problem
Cause - what causes the problem
Candidate solutions - possible solutions to the problem and their trade-
offs
Decision - the solution chosen
Rationale - why the solution was chosen, e.g., an architectural decision
record
The chosen solution modifies the architecture. It adds, deletes and modifies
elements and relations. A solution can also add, delete and modify design
rules and constraints, which influence future design decisions. The design
rules and constraints restrict how elements can be combined and interact.

---

## Page 99

Figure 3.17 An architecture is the composition of design decisions, each of which consists of the
problem, motivation, cause, candidate solutions, chosen solution, and rationale. The chosen
solution modifies the architecture by adding, deleting and modifying elements, relations, rules,
and constraints.
Let’s imagine, for example, that a design decision chooses the layered
architecture, which I described in Section 3.3.1. The solution consists of the
layers (elements), such as web/api, domain, and infrastructure, along
with rules/constraints that specify that all elements must belong to one of the
layers and that a layer can only depend on the layer immediately below it.
Initially, the architecture is a blank sheet. The first design decision adds the
initial elements, relations, rules and constraints. Each subsequent design
decision that’s then made modifies the architecture. The architecture is a
composition of all of those design decisions.
This design decision oriented view of software architecture has several
benefits. The first is that each architecture element can be traced back to a
design decision. In other words, it should exist if and only if it solves a
problem. Keeping this principle in mind can help avoid over-engineering.
Another benefit, which is one of the main motivations for the paper, is that it
preserves the rationale behind the architecture. Normally, the rationale is lost
as the architecture evolves - a phenomenon that the authors refer to as
knowledge vaporization. This approach encourages the preservation of the
rationale behind each design decision by explicitly documenting it.
Now that we have seen how architecting is a process of making architectural
design decisions, let’s look at who does it.

3.4.2 Who defines an application’s architecture?
People, of course, make architectural decisions. But the key question is:
which people? Are decisions made by developers, or are they responsibility
of architects? And, if architects are involved, are they embedded in the teams
or, do they operate as a separate group?
When deciding who is responsible for architecture, there are several factors
you need to consider:
Quality - architectural decisions must satisfy the needs of everyone
impacted by those decisions
Speed - in the era of fast flow, architectural decisions must be made
quickly
Buy-in - the people impacted by the decisions must buy into them

3.4.3 When to architect?
Gall’s Law is a rule of thumb for systems design from his book
Systemantics: How Systems Really Work and How They Fail. It states:
A complex system that works is invariably found to have evolved from a
simple system that worked. A complex system designed from scratch
never works and cannot be patched up to make it work. You have to
start over with a working simple system.
The implication of Gall’s Law is that you start with the simplest architecture
that works. You then evolve the architecture as the application grows.
Similarly, from the perspective of the Architecture as a Set of Architectural
Design Decisions paper I mentioned earlier, this means that initially you
only make design decisions to solve the problems that truly need to be
solved. And then, as your application grows, and you encounter new
problems that need to be solved, you make design decisions that solve those
problems.

scenario, while you don’t currently have a problem, there’s actual evidence

Now that you have learned about the fundamentals of software architecture,
let’s look at one of the most important architectural principles: loose
coupling. Loose coupling is so central to good architecture and design that it
deserves a deep dive. That’s because an architecture must be loosely coupled
to satisfy key quality attributes, such as availability and ease of change.
Without it, the application will most likely be difficult to change and prone
to failure. If you have ever spent endless meetings trying to coordinate
changes with other teams, then most likely you have experienced the pain of
tight coupling.
To understand why loose coupling is important, we need to first define what
is meant by coupling. Two software elements are coupled if one can affect
the other in some way. Coupling manifests itself in various ways. In this
chapter, I cover four different types of coupling. Two types of coupling -
design-time coupling and build-time coupling - affect how quickly teams can
develop and deploy software and so directly impact fast flow. The remaining
two types - runtime and infrastructure coupling - determine the application’s
runtime behavior, most notably, its availability and latency. Let’s start by
exploring the concept of coupling.
4.1 Introduction to coupling
Let’s start with a very abstract definition of coupling, which will refine later:

---

## Page 107

Coupling is a relationship between pairs of software elements that
represents the ability of one element to influence the other element in
some way.
It’s extremely desirable for elements to be loosely coupled, which means that
one element has little or no influence on the other element. The opposite of
loose coupling is tight coupling, where one element has a significant
influence on the other element. Tight coupling is the cause of numerous
development and operational issues, and is best avoided.
But what on earth do the phrases "the ability of one element to influence the
other element", "loosely coupled" and "tightly coupled" actually mean? To
make sense of these terms, it’s important to understand the different types of
coupling.
4.1.1 Four flavors of coupling
As figure 4.1 shows, there are four different types of coupling.
Design-time coupling - the likelihood that a change to one element will
require a change to the other element.
Build-time coupling - when changing one element requires another
element to be recompiled and retested.
Runtime coupling - when one element’s failure causes another element
to fail.
Infrastructure coupling - when two elements share infrastructure, such
as a database, and one element’s behavior affects the other element

4.1.2 Why loose coupling matters
Loose design-time and build-time coupling are essential for fast flow. By
localizing changes, loose design-time coupling enables teams to work
independently and in parallel. It also makes it easier for a team to make
changes. Loose build-time coupling is also important, especially for large
code bases, because it accelerates the deployment pipeline.
The other two types of coupling are unrelated to fast flow. Instead, they
affect the application’s runtime behavior. If a service is tightly runtime
coupled to another service, then the failure of the second service will cause
the first service to fail, which reduces availability. Tight runtime coupling
can also increase latency since it introduces additional network hops.
Infrastructure coupling can also reduce availability. If two services share
infrastructure, then the behavior of one service could affect the other by, for
example, stealing resources.

4.2 Design-time coupling
The first type of coupling is design-time coupling. As I mentioned in chapter
1, loose design-time coupling is a defining characteristic of the microservice
architecture and a key reason why the architectural style supports fast flow.
The services - i.e., components - must be loosely design-time coupled, and
so a change to one should rarely affect another. Importantly, design-time
coupling isn’t just a relationship between the components of the component
view. It’s also a relationship between the elements of the domain view. The
classes, packages, and subdomains of the domain view must also be loosely
design-time coupled.
In this section, I’ll explain the concept of design-time coupling and why
loose design-time coupling is essential. You will also learn about cohesion, a
closely related concept. I’ll describe why it’s important to design cohesive
software elements, which are elements whose child elements are tightly
coupled. After exploring these foundational concepts, we will look at the
benefits of loose design-time coupling and how to achieve it.
4.2.1 Design-time coupling and cohesion
An integral part of software design is organizing software elements into a
hierarchy that minimizes design-time coupling, whenever possible, and
manages it when unavoidable using the related concept of cohesion. At the
lowest level, methods and properties are organized into classes. Those
classes are then organized into packages, which can be further grouped into
higher-level packages. Beyond this, packages are organized into
subdomains. Finally, subdomains are organized into components. Each level
of hierarchy is shaped by the twin goals of minimizing design-time coupling
and maximizing cohesion. Let’s look more closely at each of these concepts
starting with design-time coupling.

ABOUT DESIGN-TIME COUPLING
Loose design-time coupling is an ancient concept. For example, the famous
computer scientist David Parnas wrote a famous paper about modularization
back in 1972. The paper’s title is "On the criteria to be used in decomposing
systems into modules" (https://dl.acm.org/doi/10.1145/361598.361623). The
key idea was to design modules in a way such that a change to one module
would have minimal impact on other modules. Another early paper called
"Structured Design" was written in 1974 by Larry Constantine described the
concepts of coupling and cohesion. Over fifty years later, the concepts
described in these papers are still extremely relevant.
Design-time coupling refers to the likelihood that a change to one software
element will require changes to another. Some design-time coupling in an
architecture is inevitable. For example, the Order Management subdomain
uses the Delivery Management subdomain to schedule a delivery for an
order. It is, therefore, coupled to the Delivery Management subdomain.
There’s a risk that a change to Delivery Management will require a change
to Order Management. The trick is to design these two subdomains in a way
that they are loosely design-time coupled: future changes to Delivery
Management rarely affect Order Management.
As I describe below in more detail in Section 4.2.2, loose design-time
coupling is important because it makes change easier and safer. If two
subdomains are loosely design-time coupled, then a developer can change
one without having to consider the other. Not only does the ability to make
localized changes improve the life of an individual developer, but if the
subdomains are owned by different teams, then it enables them to work
independently and in parallel. The delivery team can, for example, change
the Delivery Management subdomain without having to coordinate with the
orders team.
ABOUT COHESION
Cohesion is a concept closely related to design-time coupling. It’s a principle
that organizes the hierarchy of software elements based on how tightly they
are design-time coupled. A software element is cohesive if its child elements

---

## Page 111

are tightly design-time coupled. Those child elements that are not tightly
design-time coupled should belong to a different parent element. A parent’s
child elements are tightly design-time coupled to one another but loosely
design-time coupled elements with other parents.
This definition of cohesion might sound surprising since tight design-time
coupling is so undesirable. But in reality, some amount of tight design-time
coupling is unavoidable. And the best way to manage it is for the tightly
design-time coupled software elements to have the same parent element. For
example, classes that are tightly coupled should be in the same package.
Similarly, tightly coupled packages should belong to the same parent
package. Tightly coupled subdomains should be in the same component.
Organizing tightly coupled software elements in this way will ensure that
changes that inevitably affect multiple elements are still as localized as
possible.

example, different services could use different versions of a shared utility.

4.3 Build-time coupling
The second type of coupling is build-time coupling, which is a type of
relationship between the subprojects of a build project. A build project,
which is part of the repository and deployment dimension of architecture
discussed earlier in chapter 3, organizes the source code for one or more
subdomains into build subprojects that build and test components. For
example, the build project for the FTGO monolith consists of multiple
subprojects, such as order-management, customer-management, and
delivery-management. A build subproject, such as order-management, is
build-time coupled to build subproject customer-management if changing
customer-management requires order-management to be recompiled and
retested even though it hasn’t changed.

---

Like design-time coupling, build-time coupling also affects development.
But rather than influencing how easy it is for a developer to make a change,
build-time coupling determines how long the developer who has made a
change must wait for feedback. Ideally, only the changed subprojects should
be recompiled and retested, which is typically much faster than recompiling
and retesting the entire component. Build-time coupling, however, increases
the number of subprojects that must be recompiled and retested. In the worst
case, the developer might need to wait for the build tool to recompile and
retest the entire component!
To better understand the concept of build-time coupling, let’s look at a
concrete example. There are many build tools, but my favorite build tool is
Gradle (https://gradle.org/). It’s the one that I’m going to describe here,
although many others work similarly. I’ll first describe build-time coupling
in a Gradle project. After that, I’ll describe a couple of ways to minimize it.
4.3.1 About build-time coupling in Gradle projects
A Gradle build project, which builds and tests components, usually consists
of multiple subprojects. You typically define dependencies between
subprojects

classes at either compile-time, runtime or both. Because Gradle implements
incremental builds
(https://docs.gradle.org/current/userguide/incremental_build.html#header), it
will only recompile and retest those subprojects that are affected by a
change: the subproject that’s changed and those subprojects that depend on
it. It’s these dependencies that determine build-time coupling between
projects.

4.4 Runtime coupling
The third type of coupling is runtime coupling. Unlike, design-time coupling
and build-time coupling, which impact development, runtime coupling
affects the runtime behavior of a distributed application. In a distributed
architecture, it’s common for an operation (aka request) to span multiple
components. A component, such as the Order Service, is runtime coupled
to component, such as the Customer Service, if Order Service cannot
respond to request until it receives a response from Customer Service. Or,
in other words, the failure of Customer Service causes Order Service to
fail. As a result of runtime coupling, the availability of the operation depends
on the availability of both components.
Runtime coupling is one of the reasons that the famous computer scientist
Leslie Lamport
(https://amturing.acm.org/award_winners/lamport_1205376.cfm) once
wrote:
A distributed system is one in which the failure of a computer you didn’t
even know existed can render your own computer unusable.
It’s important to pay close attention to runtime coupling when designing a
distributed architecture, such as a microservice architecture. Otherwise, you
risk building an application that is fragile and unreliable.
Let’s first look at an example of tight runtime coupling and see how it
impacts the application’s availability and latency. After that, we’ll look at
some strategies for minimizing runtime coupling.
4.4.1 An example of tight runtime coupling
To understand how tight runtime coupling reduces availability and increases
latency, let’s consider an example. Figure 4.12 shows a simple architecture
where the Order Service handles a POST /orders request by invoking the
Customer Service to validate the customer and reserve credit. The Order
Service is runtime coupled to the Customer Service because it cannot

---

respond to the POST /orders request until the Customer Service has
responded to the PUT /customers/{id} request.
Figure 4.12 The Order Service cannot respond to the POST /orders request until the Customer
Service has responded to the PUT /customers/{id} request. As a result, the availability of the
POST /orders request is the product of the availability of the Order Service and the Customer
Service
Since both services must be available in order for the application to handle
the POST /orders request, the availability of that endpoint is reduced.
Mathematically speaking, the availability of the POST /orders request is the
product of the availability of the Order Service and the Customer Service.
If the Customer Service is 99.9% available and the Order Service is
99.9% available then the availability of the POST /orders request is 99.8%.
While even a chain of two services reduces availability, a common anti-
pattern is to have long chains of numerous services that are runtime coupled.
They behave like Christmas tree lights where a failure of an individual
service causes the entire chain to fail.
Tight runtime coupling can also increase latency since the response time is
the sum of the request response times. For example, the response time of the
POST /orders request is the sum of the response times of the Order
Service and the Customer Service.
4.4.2 Siloed organization can lead to tight runtime coupling
Tight runtime coupling often arises in organizations where no one is paying
attention to the end-to-end flow of requests. For example, while conducting

services, which then invoked Team B’s service. From Team A’s perspective,
everything seemed fine. But when I talked to Team B, I discovered that they
had implemented a cluster of services. As a result, the request was dependent
on the availability of many services. There was a risk that the client could
not meet their SLA because of this runtime coupling.

There are few different strategies to minimize an operation’s runtime

Implement the operation using asynchronous messaging

4.5 Infrastructure coupling
The fourth type of coupling is infrastructure coupling, and like runtime
coupling, it affects the application’s runtime behavior. An application’s
components run on and consume infrastructure. Infrastructure is a broad
term and includes the basic compute and networking resources, along with
services, such as database servers and message brokers. Components that use

It’s important, therefore, to minimize infrastructure coupling, especially for
highly available components. Let’s first look at an example of infrastructure
coupling. After that, I’ll describe strategies for minimizing infrastructure
coupling.
4.5.1 An example of infrastructure coupling
To understand why infrastructure coupling is important, let’s consider once

COMPETITION FOR RESOURCES
The first type of infrastructure coupling is competition for resources.
Kubernetes lets you configure limits on the CPU and memory resources that
a container can consume. Provided that you remember to configure limits,
the Customer Service cannot, for example, starve the Order Service of
those kinds of resources. However, it could steal all the available network

The second kind of infrastructure coupling is related to configuration
changes. Complex systems often fail when a configuration change has
unintended consequences. Consequently, in the Customer Service and
Order Service example, there’s a risk that a configuration change to the
Kubernetes cluster or the database server for one service could accidentally
affect the other service.
4.5.3 Minimizing infrastructure coupling
We can minimize the infrastructure coupling of truly critical services
through a combination of service design and deployment infrastructure
strategy. First, we should design such services so that they are runtime
decoupled from other services. This will ensure the services are unaffected
by the failure of other services. Second, we should deploy the critical
services on their own infrastructure with separate infrastructure services.
This ensures that the critical services are not competing for resources and are
not affected by configuration changes made to support other services. By
applying these principles, we can increase the availability of critical services.

4.6 Summary
Two software elements are coupled if one can affect the other in some
way either at development time or runtime
There are four types of coupling:
Design-time coupling - the likelihood that a change to one element
will require a change to the other element.
Build-time coupling - when changing one element requires another
element to be recompiled and retested.
Runtime coupling - when one element’s failure causes another
element to fail.
Infrastructure coupling - when two elements share infrastructure,
such as a database, and one element’s behavior affects the other
element
Loose design-time coupling, which localizes changes by minimizing
the number of software elements impacted by a change, is essential for
fast flow since it increases team autonomy and makes changes easier to
implement.
Loose build-time coupling, which reduces the number of build
subprojects that must also be recompiled and retested when a given
build module is changed, is important for a fast deployment pipeline,
especially for large codebases.
Loose runtime coupling is important for high availability and low
latency since it reduces the likelihood that a failure or slowness of one
component will cause another component to fail or to be slow.
Loose infrastructure coupling is essential for high availability since it
reduces the likelihood that one component will affect another
component’s runtime behavior by competing for resources or by
configuration changes.
While it’s generally desirable to minimize the different types of
coupling, it’s important to remember that some degree of coupling is
inevitable, and eliminating coupling might have significant drawbacks.

5.4.3 Testable using automated tests
Automated testing is one of the main foundations of modern software
development. Manual testing is typically too slow and error-prone to support
the volume of changes in a fast flow organization. Manual testing, especially
when done by a separate QA team, slows down development. Developers
need to run automated tests while doing development. Moreover, when a
developer pushes a change, they need feedback faster that manual testing can

---

provide. It’s essential, therefore, that developers write and run automated
tests that are sufficient to determine releasability.
An application’s architecture must make it easy for developers to write
automated tests. In fact, a useful concept is Design for testability: the design
of the software should enable easy testing. There are three important design
strategies that make testing easier:
Architect small, loosely design-time coupled elements
Eliminate or limit non-determinism
Use patterns such as dependency injection and functional-style
programming
Let’s look at each one in turn.
SMALL, LOOSELY DESIGN-TIME COUPLED ELEMENTS
The first design strategy is to structure the architecture as a collection of
small, loosely design-time coupled elements. Simple elements are easier to
test than complex elements because they require fewer, and simpler tests.
Elements with stable APIs are easier to test than those with unstable APIs,
since the tests, which are API clients, need to be updated less often.
ELIMINATE OR LIMIT NON-DETERMINISM
The second design strategy is to eliminate, or, at least limit, non-
determinism. In order to be verifiable using automated tests, the application
should be deterministic. Non-deterministic code can cause tests to be flakey,
so should be separated from deterministic code. Ideally, the non-
deterministic code should be in its own component with its own, sometimes
flakey, deployment pipeline.

DEPENDENCY INJECTION AND FUNCTIONAL-STYLE
PROGRAMMING
The third strategy is to use design techniques, such as dependency injection
and functional-style programming. Dependency injection simplifies testing
by allowing troublesome dependencies to be replaced with a test doubles.
For example, if a software element has a dependency on an external service
that complicates testing, a test could be simplified by injecting the element
with a test double that simulates the service.
Functional-style programming structures the code as side effect free
functions that have no dependencies. This approach greatly simplifies
testing, as a test simply invokes a function and makes assertions about the
return value. Consequently, there is no need to configure and inject test
doubles.
5.4.4 Testable using fast running tests

developer implements a flow item, they change one or more source files in
one or more build projects. To validate a change, one option is to execute the
test suite for every build project. However, for a large application, this could
be too slow or require significant and expensive testing infrastructure.
A better and faster approach would be to execute the test suites of only the
build projects that are affected by the change, as shown in figure 5.14. The
affected projects include not only the build projects that contain the changed
files, but also build projects that depend on them.

INCREMENTAL BUILDS
One key requirement is a build tool that supports incremental builds. A build
tool should only recompile and retest those build projects that are affected by
change rather than wasting time retesting those parts of the application that
are unaffected by the change. For example, as I described in chapter 4,
Gradle implements incremental task execution. It will only re-execute a test
task that previously succeeded if either its test sources or its test classpath
have changed. As a result, Gradle only retests the build projects containing
the changed files and other projects that have those projects on their test
classpath. This usually takes much less time than retesting everything.
LOOSE BUILD-TIME COUPLING
Incremental task execution can dramatically accelerate the tests. We can
further speed up testing by minimizing build-time coupling - a type of
coupling, which I described in chapter 4. While a build project that contains
changed files must obviously be tested, carefully organizing source files and
build projects can reduce build-time coupling, and minimize the number of
dependent projects For example, as shown in figure 5.15, rather than having
a subdomain’s clients depend on its frequently changing implementation
build project, they should instead depend on the more stable API build
project. This approach ensures that the client projects don’t need to be
retested when the implementation changes.

5.4.5 Running testing locally
Not only must tests be fast, but a developer should not have to use a separate
test environment unless, as I describe below, it’s absolutely necessary. A
remote testing environment should be avoided because shipping changes to
it slows down the developer’s edit-test-analyze loop. Local testing also
avoids the overhead and complexity of initializing a possibly shared, and
hence unstable, remote testing environment to a known state. In fact, the
Accelerate authors have identified the ability to test locally as a key
capability of high performance development.
The application’s component architecture plays a major role in whether local
testing is practical. If the application is small, then the developer can run the
entire application and its tests on their laptop. But if the application is large
and depends on many infrastructure services, a developer’s laptop might not
have enough resources to run the tests quickly.
There are a couple of solutions. One option is to buy larger laptops for the
developers. Another option is for the application to have a multi-component
architecture where each component can be developed and tested
independently. Let’s look at each option in turn.

BUY BIGGER LAPTOPS

5.5 Deployability: ease, speed and reliability of
deployments
Deployability is a quality attribute that along with testability determines the
speed of the deployment pipeline. Specifically, it describes how quickly,
easily and reliably an updated component that’s been tested by the pipeline
can be deployed into production. In a fast flow organization, changes are
deployed at least once per developer per day, and so it’s important for an
application to have high deployability.
Deployability defines the requirements for the deployment mechanism that
is triggered by the deployment pipeline after it has validated a change. The
deployment mechanism is responsible for deploying the updated component
into production. The details of the deployment mechanism including whether
it is part of the deployment pipeline or separate from it, and how it’s
triggered depend on the deployment technology used. For example, as I
describe in chapter {chapter-kubernetes}, the workflow to deploy a Spring
Boot application into a Kubernetes cluster consists of the following steps:
1. Building a container image containing the application and publishing it
to a container registry
2. Updating the production environments Kubernetes YAML to reference
the newly published version of container image
3. Triggering a rolling update of the Kubernetes Deployment, which
replaces the old service instances with new services instances
In that chapter, I describe a deployment mechanism that implements that
workflow. This mechanism is split into two parts: one that runs in the
deployment pipeline, and another that runs in the Kubernetes cluster.
Let’s first look at quality attribute scenarios for deployability. Afterward, I
will describe how an architecture can achieve high deployability.

5.5.1 Deployability quality attribute scenarios
Here are several quality attributes scenarios that capture deployability
requirements. The first scenario describes how an updated component is
quickly deployed into production without any downtime:
A deployment pipeline finishes testing a component and triggers a
deployment. The updated component is deployed into production within
5 minutes. No SLAs are violated.
The second scenario describes when a configuration bug is detected in
production and the update is rolled back:
A deployment pipeline finishes testing a component and wants to deploy
it. The updated component is deployed into production but fails to start
due to a configuration problem. The startup failure is detected, and the
deployment is rolled back. No SLAs are violated.
The third scenario describes when a performance problem is detected in
production and the update is rolled back:
A deployment pipeline finishes testing a component and wants to deploy
it. The updated component is deployed into production, but REST
endpoints exhibit unacceptability high latency because of an optimized
query. The problem is detected and the deployment is rolled back. No
SLAs are violated.

AUTOMATED DEPLOYMENTS
After a deployment pipeline has tested a component, a human could
manually update production using either the command line or a GUI
console. However, manual deployments are tedious, slow and error-prone.
It’s unlikely that a human could keep up with the continuous stream of
changes that are typical in a fast flow organization. They would become an
obstacle to fast flow. And what’s worse, they are likely to make mistakes.
A much better approach is to implement the DevOps technical practice of
automated deployments. Specifically, the deployment pipeline should
automatically deploy the updated component into production. Automated
deployments are faster and more reliable than manual deployments. They
also reduce a team’s cognitive load since they don’t need to remember the
steps required to deploy the component.
